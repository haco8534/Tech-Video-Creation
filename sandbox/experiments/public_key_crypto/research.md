# 公開鍵暗号 — リサーチ

`script.md` の前段。公開鍵暗号について、台本の**正確さ**と**切り口**を支えるために調べた記録。

## 主流（基本の仕組み）

- 公開鍵暗号は、**暗号化と復号に別々の鍵を使う**方式。受け手が「公開鍵」と「秘密鍵」を対で生成し、公開鍵は広く配り、秘密鍵は自分だけが持つ。送り手は公開鍵で暗号化し、受け手は秘密鍵で復号する。1976年に Diffie と Hellman が概念を公表し、1977年の RSA が具体的な実装を与えた。
- 安全性は **一方向性**（行きはやさしく帰りは困難な計算）と **落とし戸（trapdoor）** に依る。RSA は「大きな二つの素数の掛け算はやさしいが、その積を素因数分解するのは現実的な時間で困難」という非対称性に依拠する。秘密鍵は、困難な帰り道に付いた近道にあたる。
- 同じ鍵の対は **逆向きにも使える**。秘密鍵で施した処理を公開鍵で誰でも検証できる——これがデジタル署名で、送り主の真正性と改ざんの検知を担う。
- 実際の HTTPS などは、公開鍵暗号で高速な「共通鍵」を安全に渡し、本文の暗号化自体は共通鍵暗号で行うハイブリッド方式が一般的。台本では本文も公開鍵で運ぶ単純化した形で語り、骨格を見せる。

## 異議（通説とのズレ・限界）

- **「絶対に解けない」のではなく「現実的な時間では解けない」。** 計算量的な安全性であり、鍵が十分大きくないと破られる。安全とされる鍵の大きさは時代とともに見直されている（RSA は 1024bit が非推奨化し 2048bit 以上へ）。→ 台本でも「絶対」と言い切らず、補足で相対化する。
- **速い計算機が現れれば話は変わる。** 大規模な量子コンピュータが実用化すると素因数分解が高速に解けうるため、耐量子の暗号方式の標準化が進んでいる。→ 台本では「うんと速い計算機」と軽く触れ、議論を閉じない。
- **RSA は一例にすぎない。** 楕円曲線など、別の難問に基づく公開鍵暗号もある。「素因数分解」を唯一の土台のように語らない（「たとえば」と添える）。
- **鍵を配るだけでは「その公開鍵が本当に相手のものか」は保証されない**（なりすましの余地）。実運用では認証の仕組みが要る。台本はここまで踏み込まず、骨格に絞る。

## 背景

- 出発点は **鍵配送問題**。一つの鍵で閉めも開けもする方式（共通鍵暗号）では、暗号で話す前に、その鍵を安全な経路で相手へ渡さねばならない。だが安全な経路があるなら、そもそも苦労しない——という堂々めぐり。
- 日常の接点：ブラウザの鍵マーク（HTTPS）、ネット決済、メッセージアプリの暗号化、ソフトウェア更新の署名。視聴者の多くは「番号を打って送信する」体験を入口に持つ。

## 切り口

調べて引っかかったのは、暗号の核心が **「計算の複雑さ」ではなく「閉める操作と開ける操作を、別々の手に分けられること」** だという点。錠前は世界中に——盗み見する相手にすら——配ってよい。秘密なのは「開ける」行為のほうだけ。

**鍵を一つの「閉めも開けもする道具」と思う先入観を外す**ところから入る。閉めるのに鍵は要らない錠前があり、それを配ってしまえば、鍵を一度も渡さずに秘密が送れる——と運ぶ。

序論で「鍵を渡さずに、どうやって秘密を送るのか」という不思議を立て、結論で「渡す必要は無かった——閉めることと開けることは、はじめから別の手の仕事だった」と回収する。
