# public_key_crypto — design spec

`design_overall.md` の質的な動きを event アンカーに分解し、`script.md` の各セリフへ割り当てる。座標値・フレーム数は持たない。

対話字幕は event ではない——全セリフが自動で画面下部の字幕になる（tsx の仕事）。ここで定義する event は、絵の装置の挙動だけを指す。

## 1. event 一覧と割り当て

発生順。画面切替 event（`scene.*.in`）は各画面の先頭セリフに必須。

| event | 紐付くセリフ（画面・話者・抜粋） | 発火する挙動 |
|---|---|---|
| `scene.intro.in` | 序論・めたん「ネットでお買い物するとき…」 | 序論の画面がフェードイン。通信路と送り手・受け手の台が現れる |
| `intro.channel` | 序論・ずんだもん「…長い通信路が伸びているのだ。…」 | のぞき見る目が通信路の上に現れる。開いた箱が通信路を進み、目が中の手紙をコーラルに染める |
| `intro.naive` | 序論・ずんだもん「すなおな考えなのだ。箱に入れて、錠前をかけて、送る。」 | 素朴な錠前と鍵が現れ、箱のふたを閉じて錠前をかける（施錠度0→1）。閉じた箱は中身を読まれない |
| `intro.keyleak` | 序論・ずんだもん「…のぞいている人に、鍵まで見られるのだ。」 | 鍵が通信路を進み、目がそれをとらえてコーラルに染まる。漏れた鍵で箱が開きコーラルの✗が灯る |
| `intro.name` | 序論・ずんだもん「その仕掛けに、名前があるのだ。公開鍵暗号、というのだ。」 | 通信路上に錠前のかかった箱がとどまり、その下にタイトル「公開鍵暗号」が灯る |
| `scene.body1.in` | ボディ1・ずんだもん「まず、思いこみをひとつ外すのだ。…」 | 画面切替。通信路と台が現れ、中央に一つの南京錠（つるが開いた状態）が立つ |
| `b1.snap` | ボディ1・ずんだもん「パチンと押しこめば、それだけで掛かる錠前があるのだ。…」 | 中央の南京錠のつるが押しこまれパチンと閉じる（鍵は使われない） |
| `b1.pair` | ボディ1・ずんだもん「受け手が、錠前と鍵をひとそろい用意するのだ。…」 | 南京錠の脇に対の鍵が現れ、鍵が受け手の手元ゾーンへ収まる（秘密鍵） |
| `b1.publish` | ボディ1・ずんだもん「いっぽうの錠前は、開いたまま、いくつも通信路に流してしまうのだ。」 | 開いた錠前が複製され、通信路を右から左へ何個も流れる。目もティール色で一つ受け取る |
| `b1.lock` | ボディ1・ずんだもん「送り手は、配られた錠前で箱をパチンと閉じて、送り返すのだ。」 | 送り手が錠前を一つ取り箱のふたを閉じる（施錠度0→1）。箱が通信路を右へ進む |
| `b1.open` | ボディ1・ずんだもん「手元の鍵で、かちゃり。鍵そのものは、一度も通信路を通っていない…」 | 受け手の手元の鍵が箱の錠前に差さりふたが開く（施錠度1→0）。グリーンの✓が灯る |
| `scene.body2.in` | ボディ2・めたん「でも、ひとつ引っかかるの。…」 | 画面切替。中央に大きな南京錠が一つ立つ |
| `b2.inside` | ボディ2・ずんだもん「…錠前の中身は、金属ではなく、ひとつの計算なのだ。」 | 南京錠の金属の殻が左右に開き、中の計算盤があらわになる |
| `b2.forward` | ボディ2・ずんだもん「たとえば、大きな素数をふたつ、掛け合わせるのだ。…」 | 左の二つの素数タイルが中央へ寄り掛け合わさって右に積タイルになる。行きの矢印がすっと通る |
| `b2.backward` | ボディ2・ずんだもん「では逆に、掛けた答えだけを渡して、元のふたつの数を言い当てて…」 | 積タイルから帰る矢印が険しい壁の前で詰まる。総当たりの試しタイルが現れては×になる |
| `b2.trapdoor` | ボディ2・ずんだもん「…帰り道に隠し扉が開くのだ。落とし戸と呼ぶのだ。」 | 帰りの壁に落とし戸が現れ、合言葉の数（アンバー）が差しこまれて開き、近道が通る |
| `b2.caveat` | ボディ2・ずんだもん「念のため言うと、「絶対に解けない」のではなく…」 | 帰りの道に砂時計が小さく添えられ、「現実的な時間では」であることを示す |
| `scene.body3.in` | ボディ3・ずんだもん「ここまでで、秘密は送れたのだ。…もう一芸あるのだ。」 | 画面切替。通信路が戻り、受け手の台のそばに錠前と鍵の対が現れる |
| `b3.reverse` | ボディ3・ずんだもん「できるのだ。受け手が、手元の秘密鍵のほうで、箱を閉じて…」 | 受け手が箱のふたを秘密鍵（アンバー）で閉じる（施錠度0→1。錠前ではなく鍵で閉める） |
| `b3.anyopen` | ボディ3・ずんだもん「配られた錠前——公開鍵を持つ人なら、世界じゅうの誰でも開けられる…」 | 箱が通信路を進み、配られた公開鍵を持つ受け取る人々が現れ、箱が開く（施錠度1→0） |
| `b3.sign` | ボディ3・ずんだもん「これを署名と呼ぶのだ。…今度は送り主の印になったのだ。」 | 開いた箱にグリーンの署名の印が灯り、印から受け手の秘密鍵へ一本の線が結ばれる |
| `scene.outro.in` | 結論・ずんだもん「通信路ぜんたいを、もう一度たどってみるのだ。」 | 画面切替。中央に通信路と箱、その上に空の工程アイコン帯が現れる |
| `outro.recap` | 結論・ずんだもん「はじまりは、人目のある通信路。…鍵という秘密が先に要った。」 | 工程アイコン（人目の道→鍵を分ける→片道の計算→署名）が順に灯り、箱が全工程を早送りでなぞる |
| `outro.loopback` | 結論・ずんだもん「「鍵を渡さずに、秘密を送れるのか」という不思議なのだ。」 | 通信路に錠前のかかった箱、上にのぞき見る目、傍らに「？」が灯る（序論の不思議に回帰） |
| `outro.end` | 結論・ずんだもん「鍵を「配るもの」と「隠すもの」に分けた——…」 | 受け手の手元の鍵だけが光り、通信路に「鍵の通った跡が無い」空白が示される。箱が開き締めの一文が灯る |

内容 event は20個（全112セリフの約18%）。「5〜15%目安」を上回るが、5画面それぞれが複数の独立した工程（閉める・配る・閉じる・開く／掛ける・戻す・落とし戸…）を順送りで扱い、工程ごとに専用装置を立ち上げ・畳みする題材のため。装置の登場・退場そのものが説明の本体なので、event は工程の数に比例する。

## 3. オブジェクト別 挙動メモ

複数 event をまたいで連続変化するオブジェクトの、event 間の状態の向き先。座標・フレーム数は持たない。

**箱（背骨）** — 全画面に登場。持つ数値は「施錠度」（0＝ふたが開いて中身が見える／1＝錠前で固く閉じている）と、通信路上の位置。
- 序論：画面入場で施錠度0（送り手の台）→ `intro.channel` で通信路を右へ動き、開いたまま目に読まれる → `intro.naive` で施錠度0→1（錠前がかかる）→ `intro.keyleak` で漏れた鍵により施錠度1→0（破られる）→ `intro.name` で施錠度1のまま通信路の上にとどまる。
- ボディ1：`b1.lock` で施錠度0→1（送り手側）し通信路を右へ → `b1.open` で受け手側に着き施錠度1→0。
- ボディ2：箱は登場しない（錠前の内部に集中）。
- ボディ3：画面入場で施錠度0（受け手の台）→ `b3.reverse` で施錠度0→1（秘密鍵で閉じる）→ `b3.anyopen` で通信路を進み施錠度1→0。
- 結論：画面入場で施錠度0 → `outro.recap` で全工程を早送り（0→1→0）→ `outro.loopback` で施錠度1（のぞき見る目の下）→ `outro.end` で施錠度1→0（受け手のもとで開く）。

**錠前（南京錠）** — ボディ1で「閉めるのに鍵は要らない」を見せ、ボディ2で殻が開いて計算盤になり、ボディ3で公開鍵として箱を開ける側に回る（画面リセットを挟むので別個体だが、視聴者には「同じ錠前」と感じさせる）。つるの開閉度を数値で持つ。ボディ1：`b1.snap` でつる開→閉。`b1.publish` で開いた姿のまま複製され流れる。ボディ2：`b2.inside` で殻の開き度0→1。ボディ3：`b3.anyopen` で箱の錠前に差さりつる開。

**計算盤（ボディ2）** — `b2.inside` で錠前の殻が開いて誕生。`b2.forward` で素数タイル2枚が中央へ寄り積タイルへ溶けこむ（行きの矢印が一息で通る）。`b2.backward` で積タイルから帰る矢印が険しい壁で詰まり、総当たりタイルが現れては×になる。`b2.trapdoor` で壁に落とし戸が開き、合言葉の数が差さるとアンバーの近道が壁を迂回して通る。`b2.caveat` で帰りの道に砂時計が添う。

**配られる公開鍵の群れ（ボディ1）** — `b1.publish` で中央の南京錠が複製され、通信路を右から左へ流れる列になる。のぞき見る目も一つ受け取るが、危険のコーラルではなく公開のティールで描く（配ってよい）。`b1.lock` で送り手が列から一つ取り、それが箱の錠前になる。

**署名の印（ボディ3）** — `b3.sign` で、箱が公開鍵できれいに開いた事実から緑のチェックが灯る。同時に印から受け手の手元の秘密鍵へ一本の線が結ばれ、「閉じられたのは秘密鍵を持つただ一人」という関係を示して保持。

## 4. 採否判断

- [x] design_overall の各画面の「動き」がすべて event に分解されている
- [x] 各 event が `script.md` の実在するセリフに紐付いている
- [x] 画面切替 event（`scene.*.in`）が各画面の先頭セリフに付いている（5画面ぶん）
- [x] 内容 event の量：20個・約18%（理由を §1 末尾に明記）
- [x] 全オブジェクトにライフサイクル契約（誕生・死・出生元・退場先）がある
- [x] 出生元・退場先がすべて既存オブジェクトか画面入退場（何もない所から湧いていない）
- [x] 座標値・フレーム数値を書いていない
- [x] 対話字幕を event 扱いしていない（字幕は SCRIPT 全行から自動描画）

## 次段への引き継ぎ

`Composition.tsx` 実装（`04_remotion.md`）へ：

- **event 一覧** → `scriptData.ts` の該当セリフに `event` を付与。各 event のセリフ開始フレームが Track のキーフレーム位置
- **ライフサイクル契約** → 各オブジェクトの opacity Track（誕生 0→1／死 1→0）と画面可視性ゲート
- **挙動メモ** → 箱（施錠度・位置）・錠前（つる開閉）・計算盤・公開鍵の群れ・署名の印の状態 Track の組み立て方針
- 座標・サイズ・フレーム数値は tsx 内で直書きし、プレビューで現物合わせする
