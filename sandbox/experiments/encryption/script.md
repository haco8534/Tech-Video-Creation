# 暗号化って具体的に何をやっているのか

# ▶ 導入

めたん：ねえずんだもん。暗号化って、結局なにをやってるの？
ずんだもん：いい質問なのだ。多くの説明は「読めない文字に変える」で終わるのだ。
めたん：だって読めない文字になるんでしょう？それ以上ある？
ずんだもん：あるのだ。むしろそこからが本体なのだ。
めたん：また難しいこと言い始めたわ。
ずんだもん：たとえば、同じ「OK」という文字でも、暗号化するたびに別のぐちゃぐちゃが出ることがあるのだ。
めたん：同じ中身なのに、結果が変わるの？
ずんだもん：変わらないと困る場面があるのだ。逆に、変わってはいけない部分もあるのだ。
めたん：読めなくするだけなら、そんな面倒いらなさそうなのに。
ずんだもん：だから今回は、暗号化って具体的に何をやっているのかを解説していくのだ。
ずんだもん：これがわかると、「AESを使ってます」だけでは安全性の説明にならない理由が見えてくるのだ。
めたん：いきなり不穏ね。
ずんだもん：最初に一般論を片づけるのだ。暗号化は、平文と鍵を入れて、暗号文を出す変換なのだ。
めたん：平文が元のデータ、暗号文が読めないデータ、鍵が戻すための秘密ね。
ずんだもん：そうなのだ。でも、その説明だけだといちばん大事なところを逃すのだ。
ずんだもん：本当の問いは「どう変えると、鍵なしでは手がかりが残らないのか」なのだ。
めたん：読めない形にする、じゃなくて、手がかりを残さない形にする。
ずんだもん：そのために暗号は、文字を隠しているのではなく、ビット列の関係を壊しているのだ。

# ▶ ボディ

## 暗号化は文字を混ぜるのではなく、ビットの関係を切る

ずんだもん：まず「文章をぐちゃぐちゃにする」という見方を捨てるのだ。
めたん：いきなり捨てられた。
ずんだもん：コンピュータの中では、文章も画像も通信も、最後は 0 と 1 の列なのだ。
めたん：そこはなんとなく知ってるわ。
ずんだもん：暗号化が相手にしているのは、日本語でも英語でもなく、このビット列なのだ。
めたん：じゃあ「こんにちは」を暗号化するんじゃなくて、「こんにちは」を表す 0 と 1 を変えるのね。
ずんだもん：そうなのだ。ここで素朴な案を考えるのだ。ビットを全部反転したらどうなる？
めたん：0 を 1 に、1 を 0 にする。読めなくはなりそうね。
ずんだもん：でも規則が固定なら、もう一回反転すれば戻るのだ。
めたん：あっ。隠し方そのものが分かったら終わりね。
ずんだもん：だから鍵が必要なのだ。鍵は「どのビットをどう変えるか」を決める秘密の材料なのだ。
めたん：鍵って、パスワードそのものじゃないの？
ずんだもん：実装ではパスワードから鍵を作ることもあるけど、暗号の中心にいる鍵は、決まった長さのほぼランダムなビット列なのだ。
めたん：人間が覚える合言葉じゃなくて、機械が扱う秘密の列。
ずんだもん：その秘密の列を使って、平文のビット列との関係を壊すのだ。
めたん：関係を壊すって、どういうこと？
ずんだもん：たとえば平文に「HTTP」とか「PNG」とか、よく出る形があるのだ。暗号文にその形が残ったら、そこが足場になるのだ。
めたん：読めなくても、パターンが残っていたら推理できる。
ずんだもん：そうなのだ。だから良い暗号は、平文の少しの差が暗号文の広い差に広がるように作るのだ。
めたん：1 ビット変えただけで、暗号文がガラッと変わる感じ？
ずんだもん：その感じなのだ。AES みたいなブロック暗号は、128 ビットのかたまりを、鍵で決まる巨大な並べ替えに通すのだ。
めたん：並べ替え？
ずんだもん：128 ビットの入力は、可能性が 2 の 128 乗あるのだ。その全部に対して、同じ鍵なら同じ対応表を持つようなものなのだ。
めたん：対応表って、デカすぎない？
ずんだもん：実際に表を持つわけではないのだ。何段もの置換と混ぜ合わせで、対応表を計算しているのだ。
めたん：つまり、暗号化は文字化けを作る処理じゃなくて、鍵で決まる巨大な変換をビット列にかける処理なのね。
ずんだもん：そうなのだ。ただし、この時点ではまだ落とし穴があるのだ。
めたん：まだあるの？
ずんだもん：同じ鍵で同じ 128 ビットを入れたら、同じ暗号文が出るのだ。
めたん：それ、当たり前じゃない？
ずんだもん：当たり前だけど、通信ではエグい弱点になるのだ。

## 同じものを同じ暗号文にしないため、nonce を混ぜる

ずんだもん：もし同じ注文内容が毎回同じ暗号文になるとするのだ。
めたん：中身は読めないけど、「これは前と同じ注文だ」は分かるわね。
ずんだもん：そうなのだ。暗号文そのものが指紋になるのだ。
めたん：中身を隠しても、同じかどうかは漏れる。
ずんだもん：だから現代の暗号化では、鍵だけでなく nonce と呼ばれる値を一緒に使うのだ。
めたん：ノンス？
ずんだもん：一回限りの番号、くらいに思えばいいのだ。秘密である必要はないけど、同じ鍵の下では使い回してはいけない値なのだ。
めたん：秘密じゃないのに大事なの？
ずんだもん：大事なのだ。nonce は「今回はこの回の変換を使う」と暗号に教える番号なのだ。
めたん：同じ平文、同じ鍵でも、nonce が違えば暗号文が変わる。
ずんだもん：そうなのだ。ここが、表面的な「鍵で閉じる」説明から一段先なのだ。
めたん：でも秘密じゃないなら、攻撃者にも nonce は見えるんでしょう？
ずんだもん：見えていいのだ。見えても、鍵がなければ同じ変換を作れないのだ。
めたん：鍵は秘密、nonce は公開でもいい一回番号。
ずんだもん：実際、暗号文と一緒に nonce を送ることは普通にあるのだ。
めたん：でも「使い回してはいけない」って、そんなにヤバいの？
ずんだもん：ストリーム暗号や CTR 系のモードで使い回すと、かなり凶悪なのだ。
めたん：何が漏れるの？
ずんだもん：暗号化の一つの見方は、鍵と nonce から乱数っぽい長い列を作って、平文と XOR することなのだ。
めたん：XOR って、同じなら 0、違えば 1 になるやつ？
ずんだもん：そうなのだ。暗号文は、平文 XOR 鍵ストリームなのだ。
めたん：鍵ストリーム？
ずんだもん：鍵と nonce から作った、平文と同じ長さのマスクなのだ。これを平文に重ねて、ビットごとにひっくり返すのだ。
めたん：ワンタイムパッドみたいね。
ずんだもん：近いのだ。ただし毎回本物の乱数を平文と同じ長さだけ配るのは大変だから、鍵と nonce から疑似的に作るのだ。
めたん：そこで nonce を変えれば、毎回ちがうマスクになる。
ずんだもん：そのとおりなのだ。では、同じ鍵と同じ nonce を使い回すと？
めたん：同じマスクが出る？
ずんだもん：出るのだ。暗号文1は平文1 XOR マスク。暗号文2は平文2 XOR 同じマスク。
めたん：マスクは秘密だから大丈夫じゃないの？
ずんだもん：暗号文1と暗号文2を XOR すると、マスク同士が打ち消し合うのだ。
めたん：え、消えるの？
ずんだもん：消えるのだ。残るのは、平文1 XOR 平文2 なのだ。
めたん：中身そのものではないけど、平文同士の関係が漏れる。
ずんだもん：そこから片方の文が少しでも推測できると、もう片方が剥がれていくのだ。
めたん：読めない文字が二つあるだけなのに、重ねた瞬間に骨組みが見えるのね。
ずんだもん：だから「nonce は秘密じゃないが、再利用禁止」なのだ。この一文を落とす解説は、暗号化の足元を見せていないのだ。

## ブロック暗号は、そのままでは長い文を暗号化できない

めたん：さっき AES は 128 ビットのかたまりを変換するって言ったわよね。
ずんだもん：言ったのだ。
めたん：じゃあ 1 メガバイトの画像はどうするの？
ずんだもん：そこがモードの話なのだ。ブロック暗号は部品であって、長いデータをそのまま全部は扱えないのだ。
めたん：部品。
ずんだもん：AES は、128 ビットを鍵で変換する強い歯車なのだ。でも動画、画像、通信の本文はもっと長いのだ。
めたん：だから 128 ビットずつ切って、順番に暗号化すればいいんじゃない？
ずんだもん：その素朴案が ECB モードなのだ。そして、だいたい悪い例として出てくるのだ。
めたん：なんで？
ずんだもん：同じブロックは同じ暗号文になるからなのだ。
めたん：あ、さっきの同じもの問題が、ブロック単位で起きるのね。
ずんだもん：そうなのだ。画像なら、元の形のうっすらした輪郭が暗号文側に残ることがあるのだ。
めたん：読めないのに、絵の形は残る。気持ち悪いわね。
ずんだもん：だから実際には、ブロックをただ独立に変換しないのだ。カウンタや前のブロックや nonce を絡めて、同じ平文ブロックでも違う出力になるようにするのだ。
めたん：それがモード。
ずんだもん：そうなのだ。CTR なら、nonce とカウンタを AES に入れて鍵ストリームを作り、それを平文と XOR するのだ。
めたん：AES で平文を直接変換するんじゃなくて、AES でマスクを作る使い方もあるのね。
ずんだもん：そこが大事なのだ。現代の暗号化では、AES という部品名だけでは不十分なのだ。
めたん：AES-CBC とか AES-GCM とか、後ろに付くやつ？
ずんだもん：そうなのだ。後ろのモードが、長いデータへの使い方、nonce の扱い、改ざん検出の有無を決めるのだ。
めたん：じゃあ「AESだから安全」は雑すぎる？
ずんだもん：雑なのだ。AES という強い歯車を、どんな機械に組み込むかで安全性が変わるのだ。
めたん：強い部品でも、組み方が悪いと漏れる。
ずんだもん：そして次に見るのは、さらに見落とされがちな「改ざん」の話なのだ。

## 暗号化だけでは、いじられていないとは言えない

ずんだもん：ここまでで、暗号文は読みにくくできたのだ。
めたん：やっと勝った感じがするわ。
ずんだもん：でも攻撃者は、読めなくても触れるのだ。
めたん：触れる？
ずんだもん：通信路の途中にいる相手は、暗号文のビットを反転したり、並べ替えたり、古い暗号文をもう一度送ったりできるのだ。
めたん：中身は分からないのに、いじることはできる。
ずんだもん：そうなのだ。そして暗号の種類によっては、暗号文の 1 ビットを反転すると、復号後の平文の対応する 1 ビットが反転するのだ。
めたん：えっ。読めないのに、狙って中身を変えられる可能性があるの？
ずんだもん：あるのだ。これを「暗号化は機密性の道具であって、完全性を自動ではくれない」と見るのだ。
めたん：機密性は読まれないこと。完全性は改ざんされていないこと。
ずんだもん：そのとおりなのだ。だから現代の通信では、暗号化と改ざん検出をセットで使うのだ。
めたん：そこで認証タグ？
ずんだもん：よく出てきたのだ。AES-GCM や ChaCha20-Poly1305 みたいな AEAD は、暗号文と一緒にタグを出すのだ。
めたん：タグって、封印みたいなもの？
ずんだもん：近いのだ。鍵、nonce、暗号文、そして必要なら関連データをまとめて計算した短い検査値なのだ。
めたん：関連データ？
ずんだもん：暗号化はしないけど、勝手に変えられると困るヘッダーみたいな情報なのだ。
めたん：見えていていいけど、改ざんはされたくない部分。
ずんだもん：そうなのだ。受け手は復号する前に、または復号結果を渡す前に、タグが合うか確かめるのだ。
めたん：合わなかったら？
ずんだもん：エラーにして捨てるのだ。部分的に復号した中身を返さないのが大事なのだ。
めたん：読めないようにするだけじゃなく、いじられたら開かないようにする。
ずんだもん：それが今の実用暗号の基本なのだ。暗号文は、秘密の本文と封印タグの組として扱うのだ。
めたん：じゃあ「暗号化されているから改ざんされない」は間違いなのね。
ずんだもん：間違いなのだ。正しくは「認証付き暗号として使っていれば、改ざん検出までできる」なのだ。

## 鍵は一つではなく、用途ごとに作り分ける

めたん：でもさ、鍵さえバレなければ全部いいんじゃない？
ずんだもん：その言い方も、少し粗いのだ。
めたん：また粗いって言われた。
ずんだもん：実際の通信では、ひとつの大きな秘密をそのまま何にでも使わないのだ。
めたん：どうするの？
ずんだもん：まず鍵交換で共有秘密を作る。そこから鍵導出で、暗号化用、復号用、ハンドシェイク用、アプリデータ用、みたいに用途ごとの鍵を作るのだ。
めたん：同じ秘密から、いろんな鍵を派生させる。
ずんだもん：TLS 1.3 では、この「どの段階の交通鍵でどのメッセージを守るか」がかなり厳密に分かれているのだ。
めたん：交通鍵って、通信のその区間だけで使う鍵？
ずんだもん：そう見ればいいのだ。長く使う身分証の秘密と、実際にデータを包む短期の鍵を分けるのだ。
めたん：なんで分けるの？
ずんだもん：被害を閉じ込めるためなのだ。ひとつの鍵が漏れたとき、過去も未来も全部まとめて壊れる設計は危ないのだ。
めたん：鍵にも役割分担があるのね。
ずんだもん：さらに、同じ鍵で暗号化と認証と別用途を雑に兼用すると、思わぬ関係が漏れることがあるのだ。
めたん：鍵は強いから使い回せばお得、ではない。
ずんだもん：むしろ暗号では、同じものを違う文脈で使い回すのを嫌うのだ。
めたん：nonce も使い回し禁止、鍵も用途ごとに分ける。なんだか潔癖ね。
ずんだもん：潔癖というより、関係を残さないためなのだ。
めたん：最初に言ってた「関係を壊す」に戻ってきたわ。
ずんだもん：そうなのだ。暗号化は、平文と暗号文の関係だけでなく、セッション同士、用途同士、過去と未来の関係も切り分けようとしているのだ。

## 暗号化しても、全部は隠れない

ずんだもん：ここで最後に、暗号化の限界を見るのだ。
めたん：まだ弱点あるの？
ずんだもん：弱点というより、射程なのだ。暗号化は本文の中身を隠す。でも、必ずしも通信の存在、相手、長さ、タイミングまでは隠さないのだ。
めたん：たとえば HTTPS でも、通信していること自体は見える？
ずんだもん：見えることが多いのだ。宛先の IP アドレス、通信量、いつ増えたか、どれくらい続いたか。そういう外形は別の層の情報なのだ。
めたん：中身は読めないけど、行動の影は見える。
ずんだもん：その通りなのだ。動画を見ているのか、短いメッセージを送っているのか、サイズやタイミングから推測されることもあるのだ。
めたん：暗号化って、透明人間になるわけじゃないのね。
ずんだもん：ならないのだ。本文にカーテンをかけるけど、荷物の大きさや配達先の一部は見えることがあるのだ。
めたん：じゃあ、匿名化やトラフィック隠しは別の技術が要る。
ずんだもん：そうなのだ。ここを混ぜると「暗号化しているのにバレた」という誤解になるのだ。
めたん：暗号化の仕事は、本文と鍵なし復元を切り離すこと。通信の存在を消すことではない。
ずんだもん：いい整理なのだ。そしてもうひとつ、暗号化は端点を守らないのだ。
めたん：端点？
ずんだもん：送る前のスマホ、復号した後のサーバー、画面に表示された後のデータなのだ。
めたん：そこではもう平文に戻っているものね。
ずんだもん：だから暗号化は強いけど、魔法の安全シールではないのだ。
めたん：守る場所が決まっている道具なのね。
ずんだもん：そうなのだ。本文のビット列を、鍵なしでは関係が追えない列に変える。改ざん検出を付ける。鍵と nonce の使い方を守る。そこまでが暗号化の具体的な仕事なのだ。

# ▶ 結論

ずんだもん：最初の問いに戻るのだ。暗号化って、具体的に何をやっているのか。
めたん：文字を読めない形にする、では浅かったわね。
ずんだもん：暗号化は、平文のビット列に、鍵で決まる変換や鍵ストリームをぶつけて、元の関係を追えなくするのだ。
めたん：同じ中身でも同じ暗号文にならないように、nonce で回ごとの変換を変える。
ずんだもん：長いデータにはモードが必要で、AES という部品名だけでは使い方まで決まらないのだ。
めたん：読まれないことと、いじられていないことは別。だから AEAD ではタグで封印する。
ずんだもん：鍵は用途ごとに作り分け、nonce は再利用しない。ここを間違えると、強い暗号部品でも壊れるのだ。
めたん：それでも、長さやタイミングや端点までは自動で隠れない。
ずんだもん：暗号化は、世界から情報を消す技術ではないのだ。
ずんだもん：鍵を持たない相手から、本文との関係を切り離す技術なのだ。
めたん：読めない文字の正体は、ぐちゃぐちゃではなく、関係を断ち切ったビット列だったのね。
ずんだもん：その断ち切り方を間違えないために、暗号は細かい約束でできているのだ。
