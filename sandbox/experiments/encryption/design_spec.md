# 暗号化って具体的に何をやっているのか — セリフ単位スペック

`design_overall.md` の背骨「平文との関係がどこで切れるか／どこに残るか」を、画面ごとの event に分解し、`script.md` のセリフ 1 行に紐付ける。座標値・フレーム数は下流で決める。

## 共用の装置と色言語

- **ビット列レーン** — 左＝平文、中央＝変換、右＝暗号文。レーン外の上部＝攻撃者が見える外形、下部＝鍵の手元領域。
- **平文ビット** — 読める文字からほどける 0/1 のタイル列。規則性がある部分は淡い縞で表す。
- **鍵** — ゴールドの短いビット束。画面下の手元領域から出ない。
- **nonce** — ブルーの回番号札。暗号文と一緒に右側へ送られてもよい。
- **鍵ストリーム／マスク** — シアンの長い帯。平文と同じ長さに伸び、XOR で重なる。
- **暗号文** — パターンのない濃紺のビット列。本文の縞が消える。
- **認証タグ** — グリーンの封印片。暗号文の末尾に付き、改ざん時は赤く割れる。
- **外形メタデータ** — グレーの影。宛先・サイズ・タイミングとしてレーン外に残る。

色：背景 `#F5F7FA` ／ 主文字 `#243044` ／ 平文 `#6B7280` ／ 鍵 `#C99022` ／ nonce `#3A7CC4` ／ マスク `#26A6B8` ／ 暗号文 `#243044` ／ タグ `#3E9B5F` ／ 危険・改ざん `#CC4B3C`

## 画面づくりの約束

- **箱・南京錠メタファーに逃がさない。** 一度も箱を主役にしない。すべてビット列、帯、歯車、封印片として見せる。
- **同時表示は 4 装置まで。** レーン、鍵、nonce、主役ビット列を基本にし、役目を終えた補助ラベルはすぐ畳む。
- **「関係が残る／消える」を必ず動きで見せる。** 文字ラベルで説明せず、縞・重なり・打ち消し・枝分かれの変化で示す。
- **nonce と鍵を混同させない。** 鍵は手元領域から出さない。nonce は公開レーンを流してよい。

## 画面1 導入／ビットの関係 — 文字がビット列にほどけ、鍵で決まる変換でパターンが消える

| event | 紐付くセリフ（抜粋） | 何が起きるか |
|---|---|---|
| scene.bits.in | め「暗号化って、結局なにをやってるの？」 | 白背景に文字列「OK」が現れ、レーン左に置かれる |
| text.to.bits | ず「コンピュータの中では、文章も画像も通信も」 | 「OK」が 0/1 タイル列へほどける |
| naive.flip | ず「ビットを全部反転したらどうなる？」 | タイル列が一斉反転し、同じ規則で戻せる矢印が出る |
| key.in | ず「だから鍵が必要なのだ」 | 下部にゴールドの鍵ビット束が現れ、中央の変換機へだけ線が伸びる |
| pattern.show | ず「平文に『HTTP』とか『PNG』とか」 | 平文タイルの規則部分に淡い縞が浮く |
| aes.wheel | ず「AES みたいなブロック暗号は」 | 中央に 128 ビット歯車が立ち、タイル列を呑む |
| relation.break | ず「平文の少しの差が暗号文の広い差に」 | 1 ビットの小さな点灯が右側では広い散らばりに変わる |
| same.problem | ず「同じ鍵で同じ 128 ビットを入れたら」 | 同じ入力がもう一度流れ、右側に同じ暗号文が重なって指紋として光る |

**誕生と死** — 文字列：scene.bits.in で生まれ text.to.bits で死。平文ビット：text.to.bits で生まれ aes.wheel で変換対象へ転用。鍵：key.in で生まれ動画終盤まで下部に残る。AES 歯車：aes.wheel で生まれ画面退場で死。指紋光：same.problem で生まれ画面退場で死。同時表示の最大は〈レーン・平文ビット・鍵・AES歯車〉。

## 画面2 nonce と鍵ストリーム — 鍵と nonce から毎回ちがうマスクが出る

| event | 紐付くセリフ（抜粋） | 何が起きるか |
|---|---|---|
| scene.nonce.in | ず「現代の暗号化では、鍵だけでなく nonce」 | レーン中央に鍵生成機、下に鍵、左に同じ平文が 2 本並ぶ |
| nonce.public | ず「秘密である必要はないけど」 | ブルーの nonce 札がレーン上に出て、攻撃者の目にも見える |
| mask.make.1 | ず「鍵と nonce から乱数っぽい長い列を作って」 | 鍵と nonce が生成機へ入り、シアンのマスク帯 1 が伸びる |
| xor.apply.1 | ず「平文と XOR することなのだ」 | マスク帯が平文に重なり、右側に暗号文 1 が出る |
| nonce.change | め「同じ平文、同じ鍵でも、nonce が違えば」 | nonce 札の番号だけが変わる |
| mask.make.2 | ず「nonce を変えれば、毎回ちがうマスク」 | 別模様のマスク帯 2 が伸びる |
| xor.apply.2 | ず「毎回ちがうマスクになる」 | 同じ平文から別の暗号文 2 が出て、指紋光が消える |
| roles.pin | め「鍵は秘密、nonce は公開でもいい一回番号」 | 下の鍵に「秘密」、nonce に「公開可・一回」の短ラベルが付く |

**誕生と死** — nonce：scene.nonce.in で生まれ roles.pin まで残る。マスク帯 1・2：各 make event で生まれ XOR 後に暗号文へ吸収。暗号文 1・2：各 apply event で生まれ画面退場で死。同時表示の最大は〈鍵・nonce・マスク・平文/暗号文列〉。

## 画面3 nonce 再利用の破綻 — 同じマスクを二度使うと、暗号文を重ねた瞬間にマスクが消える

| event | 紐付くセリフ（抜粋） | 何が起きるか |
|---|---|---|
| scene.reuse.in | め「でも『使い回してはいけない』って」 | 前画面の生成機が残り、nonce 札が赤い複製マーク付きで再登場 |
| same.mask | ず「同じ鍵と同じ nonce を使い回すと」 | まったく同じシアンのマスク帯が上下 2 本にコピーされる |
| two.ciphers | ず「暗号文1は平文1 XOR マスク」 | 上下の平文がそれぞれ同じマスクで暗号文になる |
| cipher.xor | ず「暗号文1と暗号文2を XOR すると」 | 二つの暗号文が中央で重なり、XOR の記号が出る |
| mask.cancel | ず「マスク同士が打ち消し合うのだ」 | シアンのマスク模様だけが消え、灰色の関係線が残る |
| relation.leak | ず「残るのは、平文1 XOR 平文2」 | 右側に平文同士の差分パターンが赤く露出する |
| guess.peel | ず「片方の文が少しでも推測できると」 | 片方に推測ラベルが入り、もう片方の一部がめくれる |
| rule.lock | ず「nonce は秘密じゃないが、再利用禁止」 | 画面下に「同じ鍵では nonce 再利用禁止」のルール板が固定される |

**誕生と死** — 再利用 nonce：scene.reuse.in で生まれ rule.lock まで残る。上下マスク：same.mask で生まれ mask.cancel で死。差分パターン：relation.leak で生まれ画面退場で死。推測ラベル：guess.peel で生まれ一拍で死。同時表示の最大は〈生成機・上下暗号文・差分・ルール板〉。

## 画面4 モードの役割 — AES の歯車だけでは長いデータを扱えず、モードがブロック列を流れに組む

| event | 紐付くセリフ（抜粋） | 何が起きるか |
|---|---|---|
| scene.mode.in | め「じゃあ 1 メガバイトの画像はどうするの？」 | 長いビット列が画面左からはみ出す |
| split.blocks | ず「128 ビットずつ切って、順番に」 | 長い列が同サイズのブロックに切られる |
| ecb.bad | ず「その素朴案が ECB モードなのだ」 | 同じ柄の平文ブロックが同じ柄の暗号ブロックに変わり、縞が残る |
| mode.in | ず「実際には、ブロックをただ独立に変換しない」 | ECB の直列が畳まれ、nonce＋カウンタの小さなレーンに置き換わる |
| counter.run | ず「CTR なら、nonce とカウンタを AES に入れて」 | カウンタ札が 1,2,3 と進み、AES 歯車からマスク片が順に出る |
| stream.stitch | ず「それを平文と XOR するのだ」 | マスク片がつながって一本のマスク帯になり、長い平文に重なる |
| mode.name | め「AES-CBC とか AES-GCM とか」 | AES の右に「GCM」「CTR」などのモード札が並び、部品名だけでは足りないことを示す |
| strong.part | ず「強い歯車を、どんな機械に組み込むか」 | AES 歯車が機械全体の一部として縮小し、モード機構が主役になる |

**誕生と死** — 長いビット列：scene.mode.in で生まれ split.blocks でブロック列へ転用。ECB 縞：ecb.bad で生まれ mode.in で死。カウンタ・マスク片・モード札：各 event で生まれ画面退場で死。同時表示の最大は〈AES歯車・カウンタ・マスク帯・ブロック列〉。

## 画面5 認証タグ — 読めないことと、いじられていないことを分ける

| event | 紐付くセリフ（抜粋） | 何が起きるか |
|---|---|---|
| scene.tag.in | ず「攻撃者は、読めなくても触れるのだ」 | 暗号文列の上に赤い手が現れる |
| bit.flip | ず「暗号文のビットを反転したり」 | 暗号文の 1 ビットが赤く反転し、復号側に歪みが出る |
| concepts.split | め「機密性は読まれないこと。完全性は」 | 画面が「読めない」と「いじられていない」の 2 レーンに分かれる |
| aead.in | ず「AES-GCM や ChaCha20-Poly1305 みたいな AEAD」 | 2 レーンが再合流し、暗号文末尾にグリーンのタグ片が付く |
| aad.in | ず「暗号化はしないけど、勝手に変えられると」 | 見えているヘッダー札がタグ計算の線にだけ入る |
| verify.ok | ず「タグが合うか確かめるのだ」 | 受け手側でタグがはまり、暗号文が平文へ戻る |
| tamper.fail | め「合わなかったら？」 | 赤い手が 1 ビット変えるとタグ片が割れ、復号レーンが閉じる |
| rule.aead | ず「認証付き暗号として使っていれば」 | 「暗号文 + タグ」をひとまとまりにした束が固定される |

**誕生と死** — 赤い手：scene.tag.in で生まれ tamper.fail で死。改ざんビット：bit.flip で生まれ tamper.fail で赤割れへ転用。タグ：aead.in で生まれ verify.ok で成功、tamper.fail で失敗例として割れる。同時表示の最大は〈暗号文・タグ・ヘッダー札・赤い手〉。

## 画面6 鍵の作り分け — 共有秘密から用途別の交通鍵が枝分かれする

|---|---|---|
| scene.keys.in | め「鍵さえバレなければ全部いいんじゃない？」 | 大きな共有秘密の種が下部に現れる |
| derive.tree | ず「そこから鍵導出で、暗号化用、復号用」 | 種から複数の細い枝が伸び、用途別の鍵札になる |

| metadata.show | ず「通信の存在、相手、長さ、タイミングまでは」 | レーン外に宛先、長さバー、時刻パルスがグレーで浮く |

## 画面8 結論 — 暗号化の仕事を「関係を切る技術」として回収する

| event | 紐付くセリフ（抜粋） | 何が起きるか |
|---|---|---|
| scene.recap.in | ず「最初の問いに戻るのだ」 | 素のビット列レーンが戻る |
| recap.transform | ず「平文のビット列に、鍵で決まる変換や」 | 平文、鍵、マスク、暗号文が一筆書きで再生される |
| recap.nonce | め「同じ中身でも同じ暗号文にならないように」 | nonce 札が回番号として差し込まれ、暗号文が変わる |
| recap.mode | ず「長いデータにはモードが必要で」 | AES 歯車がモード機構の中に収まる |
| recap.tag | め「だから AEAD ではタグで封印する」 | 暗号文末尾にタグが付き、改ざん手が弾かれる |
| recap.scope | め「それでも、長さやタイミングや端点までは」 | 外形影がレーン外へ分離して残る |
| final.cut | ず「鍵を持たない相手から、本文との関係を」 | 平文と暗号文を結んでいた薄い線が切れ、中央に「関係を切る」が灯る |
| final.still | め「読めない文字の正体は、ぐちゃぐちゃではなく」 | 文字・ビット列・暗号文が横一列に並び、暗号文だけが本文との線を持たないまま静止 |

**誕生と死** — レーン：scene.recap.in で再誕生し最後まで残る。平文・鍵・nonce・モード・タグ・外形影：各 recap event で再登場し final.still まで残る。関係線：final.cut で切れて死。同時表示の最大は〈レーン・暗号文束・タグ・外形影〉。
