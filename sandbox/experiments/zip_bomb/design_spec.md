# zip_bomb — design spec

`design_overall.md` の質的な動きを event アンカーに分解し、`script.md` の各セリフへ割り当てる。座標値・フレーム数は持たない。

対話字幕は event ではない——全セリフが自動で画面下部の字幕になる（tsx の仕事）。ここで定義する event は、絵の装置の挙動だけを指す。

## 1. event 一覧と割り当て

発生順。画面切替 event（`scene.*.in`）は各画面の先頭セリフに必須。

### 画面1（序論）

| event | 紐付くセリフ（話者・抜粋） | 発火する挙動 |
|---|---|---|
| `scene.intro.in` | め「ねえずんだもん、これ見て。画面にファイルがひとつ。」 | 序論の画面がフェードイン。中央に紙片が現れ、「42.zip」のラベルが添えられる |
| `intro.size` | ず「サイズはたったの 42 キロバイト」 | 紙片の下に「42KB」の数値ラベルが灯る |
| `intro.expand` | ず「中身を全部展開すると、4.5 ペタバイトに化ける」 | 紙片の右側に作業の流れが伸び、立方体スタックが積み上がりはじめて画面外まで伸びる。脇に「4.5PB」と「HDD 4 万 5 千台ぶん」のラベル |
| `intro.ratio` | ず「圧縮率にすると、およそ 1000 億倍」 | 紙片と立方体スタックを結ぶ線の上に「1000 億倍」のラベルが灯る |
| `intro.name` | ず「呼ばれる、有名な zip 爆弾」 | 紙片の上に「zip 爆弾」のタイトルが灯る |
| `intro.notMagic` | ず「圧縮率の魔法」じゃない」 | 「1000 億倍」のラベルが取り消し線で薄くなり、「圧縮率の魔法」ではないことを示す |
| `intro.asym` | ず「指示書は軽い、実行は重い」っていう計算の非対称」 | 紙片の脇に「指示書」、立方体スタックの脇に「実行」のラベルが灯る |
| `intro.paperEx` | ず「同じ文字を 1 億回書け」 | 紙片の表面に「同じ文字を 1 億回書け」の1行が書き起こされる |
| `intro.balance` | ず「指示書はちっちゃい、実行は天文学的」 | 紙片と立方体スタックの大きさ差が強調表示（紙片を縮め、立方体スタックの高さラベルを上に伸ばす） |

### 画面2（ボディ1：圧縮の正体）

| event | 紐付くセリフ（話者・抜粋） | 発火する挙動 |
|---|---|---|
| `scene.body1.in` | ず「正確に説明できる人は案外少ない」 | 画面切替。中央に大きな紙片が一枚立つ |
| `b1.copy` | ず「前に出てきたものを、もう一度書かない」 | 紙片の上に「コピー指示」の概念ラベルがティールで灯る |
| `b1.example` | ず「ABCABCABCABCABC」 | 紙片の上半分に「ABCABCABCABCABC」が一文字ずつ書き起こされる |
| `b1.rewrite` | ず「ABC、そしてこの 3 文字をもう 4 回くりかえせ」 | 後半 12 文字の上にコピー矢印（ティールの円弧）が引かれ、後半が「指示」に書き換わる。紙片の下に「LZ77（1977）」のラベルが現れる |
| `b1.lz77` | ず「1977 年の話なのだ」 | 「LZ77（1977）」のラベルが強調点灯（既出ならハイライト） |
| `b1.deflate` | ず「DEFLATE っていう圧縮も、この LZ77」 | 紙片の脇に「DEFLATE」の枠が現れ、その中に LZ77 が含まれる構造が示される |
| `b1.huff` | ず「ハフマン符号」を組み合わせた」 | DEFLATE 枠の中にハフマン符号タイル（「a」「q」の長短）が追加される |
| `b1.window` | ず「最大 32 キロバイトまでなのだ」 | 紙片の左に幅 32KB のスライド窓（薄い枠）が現れて滑る。「32KB」のラベル |
| `b1.compare` | ず「どっちが小さくなると思うのだ」 | 短い比較ビュー：ハフマン単独と LZ77 の棒グラフが現れる |
| `b1.gap` | ず「10 倍以上 開くのだ」 | 比較ビューの棒の長さ差が大きく開き、「10×」のラベルが灯る |
| `b1.zeros` | ず「ゼロが 10 億バイト並んだファイル」 | 比較ビューが畳まれ、紙片の上の文字列が「000…0（10 億バイト）」に置き換わる |
| `b1.cap258` | ず「最大 258 文字までしか」 | コピー指示の末尾に「258」の打ち止め印が現れ、複数並ぶ様子が示される |
| `b1.ceiling` | ず「おおよそ 1032 倍」 | 紙片の上に水平に「1 段の天井 ≒ 1032×」のラインが赤橙で引かれる |

### 画面3（ボディ2a：再帰型）

| event | 紐付くセリフ（話者・抜粋） | 発火する挙動 |
|---|---|---|
| `scene.body2a.in` | め「圧縮したものをまた圧縮する」 | 画面切替。左上に「素朴な誤解」のミニ比較ビュー（紙片を二重に包んでも棒が伸びないことを短く示す） |
| `b2a.naive` | ず「二重圧縮、三重圧縮」 | ミニ比較ビューが「縮まない」結論を出し、灰色に落として畳まれる |
| `b2a.split` | ず「古典的な「再帰展開型」と現代的な「単発型」」 | 画面中央に「再帰展開型」のラベルが灯り、これからの主役であることを示す |
| `b2a.tree` | ず「16 個の同じ zip が入っている」 | 中央に最初の zip が現れ、1 個から 16 個に枝が分かれる（深さ 1 のツリー） |
| `b2a.five` | ず「5 段重ねるのだ」 | 各枝からまた 16 個ずつ枝が伸びる動作を 5 段重ね、最下層が無数の小点に |
| `b2a.million` | め「およそ 100 万個」 | ツリーの最下層に「16⁵ ≒ 100 万」の数値ラベルが灯る |
| `b2a.leaf` | ず「おおむね 4 ギガバイトの大きさ」 | ツリー最下層から右へ作業の流れが伸び、立方体スタック（4GB の高さ）が現れる |
| `b2a.peta` | ず「ほぼ 4.5 ペタバイト」 | 4GB の立方体スタックが 100 万倍に増殖し、画面右に大きな量塊。「4.5PB」のラベル |
| `b2a.share` | ず「ぜんぶ「同じ実体」を指している」 | ツリー最下層の中で実体は 1 個だけハイライト、他は「参照」として薄く描かれる |
| `b2a.depthLimit` | ず「「圧縮ファイルを再帰的に展開する深さ」に上限」 | ツリーの上に深さの上限線（赤橙の水平線、深さ 3 のあたり）が引かれる |
| `b2a.outdated` | ず「現代の主流の対策にはもう引っかからない」 | 5 段重ねのツリーが上限線を突き抜けて引っかかる様子が示され、再帰型がアンバー色で「対策ずみ」と落ちる |

### 画面4（ボディ2b：単発型）

| event | 紐付くセリフ（話者・抜粋） | 発火する挙動 |
|---|---|---|
| `scene.body2b.in` | ず「もう一つの設計が出てくるのだ。単発型」 | 画面切替。中央に「単発型」のラベルが灯り、画面上部に空の zip 構造の帯（3 区画の枠）が現れる |
| `b2b.oneShot` | ず「1 回展開しただけで、いきなりとんでもないサイズ」 | 帯の右側に「深さ＝1」のラベルが灯る |
| `b2b.fifield` | ず「2019 年に David Fifield」 | 中央下に「Fifield, 2019」のラベルが灯る |
| `b2b.struct` | ず「「セントラルディレクトリ」っていう目次」 | 帯の 3 区画に「ローカル」「本体（圧縮データ）」「目次」のラベルが灯る |
| `b2b.trustDir` | ず「巻末の目次のほうを優先して読むのだ」 | 目次区画から本体区画への矢印が 1 本伸びる |
| `b2b.samePos` | ず「ぜんぶ同じ位置を指す」 | 矢印が 2 本、3 本、4 本……と増え、すべて本体の同じ位置を指す |
| `b2b.splits` | ず「複数のファイルとして使い回せるよう特殊な切れ目を仕込んである」 | 本体区画の中に縦線（特殊な切れ目）が複数現れ、矢印の入り口が少しずつずれる |
| `b2b.tera` | ず「10 メガバイトのファイルから 281 テラバイト」 | 矢印の出口側から立方体スタックが急成長。「10MB → 281TB」のラベル |
| `b2b.bypass` | ず「深さの上限を 1 にしても、無傷で通り抜ける」 | はるか上にある深さ上限の赤橙線が画面に入り、その下を「深さ＝1」のまま単発型が突き抜ける |
| `b2b.tools` | ず「主要な展開ツールがこの単発型をどう扱うかの一覧表」 | 右下にツール堅さの小表が現れる（3 行 × 反応列、エラー＝グリーン／律儀＝アンバー／途中＝中間色） |

### 画面5（ボディ3：律儀さが、引き金になる）

| event | 紐付くセリフ（話者・抜粋） | 発火する挙動 |
|---|---|---|
| `scene.body3.in` | ず「いちばん困るのは誰だと思う」 | 画面切替。画面左に紙片（小）、中央に実行者（顔のないアイコン、3 本バー） |
| `b3.humanOk` | ず「意外と、それでは深刻にならない」 | 紙片の脇に小さく人間のアイコンが現れ、グリーンで「気づいて止める」が示される |
| `b3.auto` | ず「自動的に中身を全部展開する側」 | 人間アイコンが薄れ、中央の実行者（自動展開する側）が強調される |
| `b3.ram` | ず「メモリの上に中身を広げてから」 | 実行者の RAM バーが伸びてMAXに達し赤くなる |
| `b3.cpu` | ず「10 億回ぶんの書き込み」 | 実行者の CPU バーが伸びてMAXに達する。続いてディスクバーも伸びる |
| `b3.av` | ず「アンチウイルスソフト」 | 実行者の上に「アンチウイルス」の小カードが現れる |

| `b3.web` | ず「ファイルアップロード機能のあるウェブサービス」 | 隣に「ウェブサービス」のカードが追加され、3 種が横並びになる |
| `b3.ironic` | ず「防ごうとする動作で自滅」 | 3 種カードと実行者がアンバー色に染まる（防御側が焼かれる構図） |
| `b3.lazy` | ず「本物のマルウェアまで通り抜けて」 | 比較として「中を見ない」設定のカードが脇に現れ、そこをマルウェアの記号がすり抜ける |
| `b3.limitSize` | ず「最初の 100 メガバイトまで読んだ時点」 | 立方体スタックを横切る赤橙の水平線（最大サイズ）が引かれ、その上の積み木が灰色に落ちる |
| `b3.limitTime` | ず「30 秒を超えたら」 | 立方体スタックの脇に砂時計（タイムアウト）が現れ、グリーンに灯る |
| `b3.ratioGuard` | ず「1000 倍を超えてたらもう中身を見ないで隔離」 | 圧縮率メーター（針付き）が現れ、針が振り切れた地点でグリーンの隔離マークが灯る |
| `b3.stream` | ず「展開しながら同時に検査する」 | 流しそうめんの樋（横長の流れ）が現れ、展開された単位が次々検査される様子。実行者バーが赤からグリーンに戻る |

合計 event：66（画面切替 6 + 内容 60）。題材が「装置の登場・退場そのものが説明の本体」なので、event は装置の数に比例する。

## 3. 次段への引き継ぎ

`Composition.tsx` 実装（`04_remotion.md`）へ：

- **event 一覧** → `scriptData.ts` の該当セリフに `event` を付与。各 event のセリフ開始フレームが Track のキーフレーム位置
- **ライフサイクル契約** → 各オブジェクトの opacity Track（誕生 0→1／死 1→0）と画面可視性ゲート
- 座標・サイズ・フレーム数値は tsx 内で直書きするが、**名前付きアンカー定数経由**で集約し、矢印・ラベル・隣接オブジェクトの基準点はすべてアンカーから derive する
- **テキストの帯**：画面ごとに y 軸を「タイトル帯／主役の図／補足テキスト帯／字幕帯（固定）」に分け、テキストはその帯内にしか置かない
