# [3] セリフ単位スペック（design_spec.md）

パイプライン3段目。前段の `design_overall.md`（と `script.md`）を入力に、`design_spec.md` を作る。design_overall で決めた**質的な動き**を、台本の**各セリフに event アンカーで時間割り付け**する段。次段 `Composition.tsx` の入力になる。

この段は新しい創造的判断を持ち込まない。overall の設計を「どのセリフで起こすか」に落とすだけ。

## この段で書くもの・書かないもの

| 情報 | 担当する段 |
|---|---|
| 各画面の登場オブジェクトと質的な動き、レイアウト構造 | [2] design_overall |
| event アンカーの定義と各セリフへの割り当て | **[3] spec ← ここ** |
| オブジェクトのライフサイクル契約（誕生・死・出生元・退場先） | **[3] spec ← ここ** |
| ピクセル座標、フレーム数値 | [4] Composition.tsx |

**design_spec は数値を一切持たない。** 座標値も書かず、フレーム数も書かない。持つのは event 名・オブジェクト名・セリフ参照・質的な挙動記述だけ。**配置は design_overall のレイアウト構造を名前で参照する**——「○○画面の HStack の右セル」「○○の VStack の3行目」のように。

## event アンカーの仕組み

**event = 名前付きの動作ビート。** design_overall の「動き」を、起こす単位に分解したもの。

- 各 event を `script.md` の**特定のセリフ1行**に紐付ける
- そのセリフの開始フレームが、次段 tsx で Track のキーフレーム位置になる
- フレーム数の計算（文字数ベース）は tsx の仕事。design_spec は「どのセリフか」だけ指定する
- **対話字幕は event ではない**。全セリフが自動で字幕になる（tsx の仕事）ので spec は関与しない。event が指すのは絵のオブジェクトの挙動だけ

design_overall の質的な動きを event に分解する例：

> overall:「記憶ヘッドが左から右へ走査し、通過したトークンの痕跡が薄れる」
> → spec の event: `head.appear`（ヘッド誕生）/ `head.move`（走査開始）/ `trails.fade`（痕跡の減衰開始）

### 命名規則

`{オブジェクト}.{動作}` で命名：`title.in` `head.move` `trails.fade` `qkv.split` `summary.in` `outro` など。題材ごとに自由に決めてよい。

### 画面切替 event（必須）

各画面（セクション）の**先頭セリフ**に `scene.{画面名}.in` を紐付ける。これが画面の入れ替わり（前画面の一掃 → 新画面のフェードイン）を起こす。画面数ぶん必ず置く。

### event の量

**3〜5セリフに1イベント、全セリフの 20〜30% が目安**（design_overall のテンポ目標と対応）。これより少ないと、画面に何も起きない時間が長くなり「止まった動画」になる。多すぎると視聴者が追えない。画面切替 event はこの割合と別枠（各画面に1つ必須）。

ミクロな動き（既存図の色変化、補助線のひと出し、輪郭の一拍強調、添字のカウントアップなど → `02_design_overall.md`「テンポ目標」）も event として数えてよい。むしろこうしたミクロ event が密度を稼ぐ主役になる。

**無音ストレッチの上限は連続3セリフ。** event が一切付かないセリフが4行以上連続するなら、その区間で必ず何かを動かす——足せる event が思いつかないなら、design_overall に戻ってミクロな動きを追加する。決して spec 段でその場ひねりの event を作らない（→ `feedback_design_spec_first`）。

## 文書の骨格

```markdown
# {番号}_{題材} — design spec

## 1. event 一覧と割り当て
発生順の表：event / 紐付くセリフ / 発火する挙動
（画面切替 event scene.X.in を各画面の先頭セリフに含める）

## 2. ライフサイクル契約
オブジェクト別の表：オブジェクト / 所属画面 / 所属レイアウト / 誕生 / 死 / 出生元 / 退場先

## 3. オブジェクト別 挙動メモ
複数 event をまたぐオブジェクトについて、event 間の連続的な状態変化を散文で

## 4. 採否判断
```

### 1. event 一覧と割り当て

全 event を**発生順**に並べた表：

| event | 紐付くセリフ（画面・話者・抜粋で特定） | 発火する挙動 |
|---|---|---|
| `scene.intro.in` | 序論・めたん「あら、ずんだもん…」 | 序論の画面がフェードイン |
| `title.in` | 序論・ずんだもん「実はその中心に…」 | title が誕生（opacity 0→1） |
| … | … | … |

挙動は質的に書く（「左端→右端へ移動」「細→太」「一斉に発光」「親 HStack の左セルへ滑り込む」）。座標値・フレーム数は書かない。

### 2. ライフサイクル契約

各オブジェクトについて、誕生から死までを event で固定する：

| 項目 | 説明 |
|---|---|
| オブジェクト | コード上の識別子（ロール名） |
| 所属画面 | このオブジェクトが生きる画面（複数画面にまたがるなら明記） |
| 所属レイアウト | design_overall の舞台で定義した親レイアウト名（例：「画面1 のメイン HStack の左セル」「ボディ2 上段の Grid」「オーバーレイ層」） |
| 誕生 | opacity 0→1 を始める event（画面と同時に出るなら「画面入場」） |
| 死 | opacity 1→0 を終える event（画面と心中なら「画面退場」） |
| 出生元 | どこから生まれるか（既存オブジェクトの近傍、または親レイアウトのセル名。画面と同時なら「画面入場」） |
| 退場先 | どこへ溶けるか（同上。画面と同時なら「画面退場」） |

**所属レイアウトを明記する意味**：そのオブジェクトの位置は、親レイアウト構造の中で決まる。tsx 段でオブジェクトを増減しても、親が自動で並べ直す——「3枚目を足したら他2枚の座標を書き直す」事故を防ぐ。**「画面のどこか」「中央付近」のような構造の外の指定は、ライフサイクル契約に書かない**。

NG の典型：

- **所属レイアウトが空欄／「画面中央」のような構造外の指定** → design_overall の舞台で定義したレイアウト構造の名前で書く。どこにも属さないオブジェクトは「オーバーレイ層」として明示する
- **出生元・退場先が「画面外」** → 同一画面内では既存オブジェクトの近傍か親レイアウトのセル名から派生させる。何もない所から湧かせない
- **「フェードアウトしてそのまま忘れる」** → 退場先を明記。後で復活するなら opacity 0 で居続けて再フェードイン
- **画面跨ぎで持ち越すオブジェクトが多い** → 画面の切り方を疑う（前段 design_overall / script へ差し戻し）

### 3. オブジェクト別 挙動メモ

複数 event をまたいで動くオブジェクト（移動・変形を伴うもの）について、event 間の連続的な状態変化を散文で書く。単純なオブジェクト（in/out だけ）は表で十分なので省略してよい。座標・フレーム数は書かない——「どの event でどの状態へ向かうか」「どのレイアウトセル間を移動するか」だけ。

## 採否判断

ドラフト後にチェック：

- [ ] design_overall の各画面の「動き」がすべて event に分解されているか
- [ ] 各 event が `script.md` の実在するセリフに紐付いているか
- [ ] 画面切替 event（`scene.X.in`）が各画面の先頭セリフに付いているか
- [ ] 内容 event の量が全セリフの 20〜30%（3〜5セリフに1イベント）に達しているか
- [ ] event が一切付かないセリフが**連続4行以上**続く区間が無いか（あれば design_overall に差し戻してミクロ event を追加）
- [ ] 全オブジェクトにライフサイクル契約（所属レイアウト・誕生・死・出生元・退場先）があるか
- [ ] 所属レイアウトが design_overall で定義された構造名か「オーバーレイ層」のどちらかになっているか
- [ ] 出生元・退場先がすべて既存オブジェクトかレイアウトセルか画面入退場か（何もない所から湧いていないか）
- [ ] 座標値・フレーム数値を書き込んでいないか

## 次段への引き継ぎ

`Composition.tsx` 実装（`04_remotion.md`）に渡すもの：

- **event 一覧と割り当て** → tsx で `SCRIPT[]` に event を付与。各 event のセリフ開始フレームが Track のキーフレーム位置
- **ライフサイクル契約** → 各オブジェクトの opacity Track（誕生で 0→1、死で 1→0）と、所属レイアウト（VStack/HStack/Grid のどのセルに置くか）、画面可視性ゲート
- **オブジェクト別 挙動メモ** → 各オブジェクトの状態 Track の組み立て方針

世界観の語彙（色・形・動き・文字方針）は design_overall.md から直接 tsx の定数へ。**レイアウト構造**（VStack/HStack/Grid の入れ子）は design_overall の「舞台」から直接 tsx のレイアウトコンポーネントツリーへ。
