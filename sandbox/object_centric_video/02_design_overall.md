# [2] 全体デザイン方針（design_overall.md）

パイプライン2段目。前段の `script.md` を入力に、`design_overall.md` を作る。**各画面に何のオブジェクトが居て・どう動くか**と、**動画全体を貫通する世界観**を、自然言語で設計する。次段 `design_spec.md` の入力になる。

**創造的なデザイン判断はこの段に集約する。** 何を語るかは [1] script で、どう見せるかはこの段で決め切る。下流の spec と tsx は、ここで決めた設計を形にするだけ。

## この段で書くもの・書かないもの

| 情報 | 担当する段 |
|---|---|
| 貫通する世界観（色・形・動きの語彙） | **[2] overall ← ここ** |
| 各画面の登場オブジェクトと動き方（質的・散文） | **[2] overall ← ここ** |
| event アンカー、セリフ単位のタイミング | [3] design_spec |
| ピクセル座標、フレーム数値 | [4] Composition.tsx |

**質的に書く。** レイアウトは「中央に横並び」「上半分に円形」のように言葉で、動きは「左から右へ走査」「一斉にフェードイン」のように言葉で。座標値もフレーム数も書かない——それは下流の仕事。

## 世界観は毎回ゼロから決める

題材ごとに以下6軸を言語化する。各軸の中身は過去作の語彙に縛られない：

| 軸 | 決めること |
|---|---|
| 貫通する比喩 | 動画全体が視聴者に与える主観的な手触り（教科書／実験ノート／工房／街歩き…） |
| 形の語彙 | 「同じ意味は常に同じ形」のマッピング。汎用の箱・矢印の寄せ集めではなく、**ひとつの仕組みをひとつ可視化する専用装置**の集まりとして設計する（→ 下の「図解は仕組みを動いて見せる」。パワポ的なカード+テキストの羅列は避ける → `feedback_slide_visual_quality`） |
| 色の語彙 | 何色運用にするか、各色に持たせる意味 |
| 動きの語彙 | 使う動き／封印する動き（ズーム・回転・パン・変形…） |
| 文字方針 | 対話字幕（話者色・位置・フォント）と、図解内に置く最小限のラベルの方針。スマホ視聴前提で大きめに（→ `feedback_slide_font_size`） |
| ナビ補助 | セクションタイトル表示・進捗表示の有無と形式（対話字幕の帯とは別に置く） |

過去作（`sandbox/experiments/`）は**事例**として参照するだけ。「毎回4色」「毎回明朝体」ではない。題材に合う語彙を毎回ゼロから組む。

## 図解は仕組みを「動いて見せる」（専用装置で組む）

この段で最も重要な判断がここにある。図解が概念に**ラベルを貼る**だけだと、見栄えはパワポになり「説明として分かりにくい」動画になる。箱に「Self-Attention」と書いて矢印でつないでも、視聴者は仕組みを理解できない。

目指すのは、**図そのものが説明になっている**状態（3Blue1Brown のような）。そのために「形の語彙」は、汎用の箱・矢印の寄せ集めではなく、**ひとつの仕組みをひとつ可視化する専用装置**の集まりとして設計する。視聴者は装置が動くのを見て、解説を聞く前に仕組みを目で理解できる。

| ラベルを貼るだけ（NG） | 仕組みを動いて見せる（OK） |
|---|---|
| 「注目」と書いた矢印 | 全長一定の帯が相手ごとの区画に分割される（配分が目で見える） |
| 「ベクトル」と書いた箱 | 数値そのものをバーの並びで持ち、計算で組み変わる |
| 「曖昧」と書いたラベル | 点が空間上でぼやけ、文脈が来て一点に収束する |

専用装置を設計する手順：

1. **背骨を一本決める** — 動画全体で姿を変えていく中心オブジェクト（例：語の「意味」を表す数値の図形）。全画面がこの一つの変化の物語になるようにする
2. **各概念に装置を一つ割り当てる** — script の各セクションの中心概念に対し、「それが起きるさまを絵で起こす」装置を1つ設計する。装置は背骨を入力に何かを計算する道具として作る
3. **装置どうしを工程としてつなぐ** — 装置はバラバラの飾りではなく、背骨を順番に加工していく一連の工程。「派手にするためだけの装置」は入れない——各装置は「これは何を可視化するか」に1対1で答えられること。答えられない装置は削る

装置の数は題材の密度しだいで増えてよい（長尺・多概念なら10以上になることもある）。判断基準は数ではなく、**各装置が固有の仕組みに対応しているか**。

```markdown

## 1. 世界観
（上の6軸を、この題材ではどう設定するか言語化）

## 2. セクション別デザイン
### 序論：{script の見出し}
- 登場オブジェクト：…
- レイアウトの考え方：…（質的に。座標は書かない）
- 動き：会話が進むにつれオブジェクトがどう変化するか
- 絵の眼目：この画面で絵だけで伝わる中心概念（1文）

### 結論：{script の見出し}

## 3. 装置カタログ
| 装置 | 可視化する仕組み | 基本の見た目 | 登場セクション |

```

`## 3` は「装置カタログ」とし、各装置に **可視化する仕組み** の列を必ず置く。説明を埋められない装置はその場で削る。

## セクション別デザインの書き方

**script.md の1セクション（1画面）= この段の1サブセクション。** script の画面リストと1対1で対応させる。各画面で4項目を埋める：

- **登場オブジェクト** — その画面に居るオブジェクトの顔ぶれ
- **レイアウトの考え方** — 空間配置を質的に（「中央に横並び」「上半分に円形配置」「左に主役・右に補助」）。座標は書かない
- **動き** — 会話が進むにつれオブジェクトがどう状態変化するか。質的に（「左から右へ走査」「太さが変わる」「一斉に発光」）。フレーム数は書かない
- **絵の眼目** — この画面で絵だけで伝わるべき中心概念。1文で

画面の入れ替わりは標準30fクロスフェード（前画面が一掃され新画面に入れ替わる）。この段では各画面の中身に集中し、境界処理は一律でよい。

### オブジェクト中心の制約を守る（→ SKILL.md 通底原則）

- 画面内ではオブジェクトが unmount せず状態を連続変化させる。「何もない場所から湧く／消える」は禁止——新しいオブジェクトの**出生元**（どの既存オブジェクトのどこから生まれるか）と**退場先**（どこへ溶けるか）を散文で書く
- 画面切替（セクション境界）だけがリセットを許す唯一の場所

### 「絵で語れるか」を検査する

各画面の「絵の眼目」が**絵だけで**示せるか確かめる。絵的手がかりが思いつかない画面は、その概念を絵で語れていない。台本側で会話に逃がすか、script.md のセクション分けを見直す（前段への差し戻し）。

### 単調を避ける

「最初から最後まで表示されているものが同じ」は失敗。各画面のオブジェクトは、その画面の会話の間に必ず状態変化する（位置・サイズ・数・色・関係線…）。変化の無い画面は設計し直す。

## レイアウトの約束（フォーマット由来の制約）

- 画面は 1920×1080。中心原点で考える（実装は viewBox `-960 -540 1920 1080`）
- 画面下部は対話字幕の帯（2行ぶん）が常駐する。主オブジェクトは上寄り〜中央に置き、下端に重要物を置かない
- スマホ視聴が主。文字もオブジェクトも大きめに、要素を詰め込みすぎない

## 採否判断

ドラフト後にチェック：

- [ ] script.md のセクション（画面）すべてに対応するデザインがあるか
- [ ] 世界観6軸が題材ごとにゼロから決まっているか（過去作の流用でないか）
- [ ] 各形が「概念にラベルを貼る箱」ではなく「仕組みを動いて見せる専用装置」になっているか
- [ ] 装置カタログの各装置が「可視化する仕組み」を1つ言えるか（無意味な飾りがないか）
- [ ] 動画全体を貫く背骨（姿を変えていく中心オブジェクト）が一本あるか
- [ ] 各画面に「絵の眼目」（絵だけで伝わる中心概念）があるか
- [ ] 各画面のオブジェクトが画面内で状態変化するか（静止画面になっていないか）
- [ ] 座標値・フレーム数・セリフ単位タイミングを書き込んでいないか（質的記述に留まっているか）
- [ ] 「何もない場所から湧く／消える」オブジェクトがないか（出生元・退場先が既存オブジェクトか、画面入退場か）
- [ ] 図解の中に説明文を書き込んでいないか（概念はオブジェクトの動きで示しているか。対話字幕はセリフ本文なので別）

## 次段への引き継ぎ

`design_spec.md` 作成（`03_design_spec.md`）に渡すもの：
