# [2] 全体デザイン方針（design_overall.md）

パイプライン2段目。`script.md` を入力に、**動画の見え方をほぼ全部**ここで決める。デザイン思想は `00_philosophy.md`。下流の spec は時間軸への割り付け、tsx は数値への翻訳——画面に何が見えるかは、ここで決まる。

このドキュメントは、読み手が各画面の止め絵と動きの段取りを頭の中で再生できるところまで連れていく。spec 段に入ったときに「これってどう見えるんだっけ？」と立ち止まらないことが、書けたかどうかの目安。

## 出力の骨格

```markdown
# {番号}_{題材} — design overall

## 1. 世界観
（色・形・動きの語彙）

## 2. 図カタログ
（各図の静的仕様、描画区分を含む）

## 3. セクション別デザイン
### {script の画面1}
- 眼目 / 舞台（レイアウト構造）/ 時間順の状態遷移 / 演出ノート
### {script の画面2}
…

## 4. 採否判断
```

ここで決めるのは「どんな絵が、どの順で起こるか」。ピクセル座標は紙の上で予測しても tsx で必ずズレるので、サイズは「主役の1/3」「画面幅の8割」「字幕帯と同じ厚み」のような相対で書ける程度の解像度に留める。タイミングも「{セリフ抜粋} の近辺」「次のセリフで」のように言葉で運ぶ——フレーム数は spec の領分。**配置は、レイアウト構造（縦並び・横並び・グリッド・重ね）で書く**——「左から○ px」のような絶対指定はしない。

## 描画方式の前提

画面は **1枚の SVG（`viewBox="-960 -540 1920 1080"` 中央原点）の上に組む**。SVG 1枚の中で、3つの道具を使い分ける：

| 道具 | 用途 | 担うもの |
|---|---|---|
| **SVG 幾何要素**（`rect` `circle` `path` `line` `polygon` `<g transform>`） | 矢印・図形・パス・グラデーション・派生計算で繋がる線 | 変形・派生・連続アニメ |
| **`<foreignObject>` で埋め込む HTML+CSS** | 文字を箱に詰める部分（カード・ラベル・本文ブロック・字幕） | 文字の折返しと箱の自動サイジング |
| **レイアウトプリミティブ**（`HStack` `VStack` `Grid` `Stack`） | 子の自然サイズから座標を派生計算する配置構造 | 配置（絶対座標を書かないで済ませる） |

**使い分けの基準**：

- **文字を内包する箱は `<foreignObject>` 側**。CSS の `display: flex` + `padding` で、文字幅に応じて箱が自動で伸びる。`measureLabelWidth` でブラウザ実測と食い違うリスクを原理的に消す
- **文字を含まない／記号としての文字だけの図形は SVG 幾何要素**。矢印の派生計算、path morph、`stroke-dasharray` の描き起こしなど、SVG でしかできない表現はここに集約
- **2つ以上のオブジェクトを並べる場面は、必ずレイアウトプリミティブで包む**。「3枚並べる」「縦に積む」「2×3グリッド」を絶対座標で書かない

この使い分けが、design_overall に **「3カード横並び」「縦2段」のような構造で舞台を書ける根拠** を与える。座標衝突・文字はみ出しは、構造で書けば原理的に起きない。

## 1. 世界観

### 固定ベース（全題材共通）

- 背景 `#f5f7fa` ／ パネル面 `#ffffff` ／ 主文字 `#243044`
- 読ませる文字は 42px 以上（具体値は `04_remotion.md`）

この3色と文字下限は、白テーマの落ち着きを担保する土台。題材を超えて共通で、出発点として既に決まっている。

### 題材ごとに組む3軸

**形の語彙** — 「同じ意味は常に同じ形」のマッピングを組む。視聴者は2回目に同じ形を見たとき、最初に何を意味したかを思い出す。形が意味を運ぶ装置になっていれば、ラベルを足す必要が減る。

**色の語彙** — アクセント2〜4色を、それぞれ動画を通して同じ意味で使い続ける。1つの色が途中で別の概念に乗り換えると、視聴者は最初に学んだ対応関係を捨てて学び直すことになる。色数を増やせば話の軸も増える——少ない色で済むほど話の構造が締まる。

**動きの語彙** — 使う動きを列挙する。動きは、状態が変わる瞬間にだけ起こって、起こり終わったら止まるのが基本形。常駐するふわふわや明滅・脈動は、本筋の変化を視覚的にかき消す背景ノイズになって、後で起こる「ここぞの動き」を目立たなくする。背景は止めておくほど、起こる動きが意味を持って届く。

## 2. 図カタログ — 各図の静的仕様

図カタログは、その図を一度説明し切るための場所。**ここを読めば、その図の絵がそのまま頭に描ける**——という状態を目指す。セクション別デザインで同じ図に触れるときは、見た目を書き直さずに名前で参照すれば済む。

各図について、以下を書く：

- **役** — 何を表すか（1文）
- **描画区分** — 次の3つから1つ。設計判断の中で最も早く決める：
  - `HTML格納器`：内部に文字・本文を含み、CSS の auto-layout に幅を任せる。例：見出しカード、ラベル付き短冊、注記ブロック
  - `SVG幾何要素`：文字を含まない、または記号としての文字のみ。例：矢印、グリッド、メーターのバー、波形、放射線
  - `合成`：外枠が幾何（変形・派生計算を要する）で、内部に文字を持つ部分だけ `<foreignObject>` の HTML を抱える。例：見出しと幾何ボディを併せ持つ複合パネル、文字付きアイコン
- **形** — 図形の種類と細部（角丸の有無、線の太さ、塗り or 中抜き）
- **サイズ感** — 画面に対する大きさ。主役なら画面高の○割、補助なら主役の○分の1、字幕帯と同じ厚み——相対で
- **色** — 世界観の語彙のどれを、どこに使うか
- **構成要素** — サブパーツと位置関係（VStack/HStack/Grid の入れ子、または「重ね」で書く）
- **登場画面** — どの画面に出るか

カタログに乗せる図は、動画を通して何度か登場するものに絞ると見通しがいい。1画面しか出てこない一発要素は、その画面の中で書き切ったほうが追いやすい。

### 描画区分の選び方

迷ったら、次の3つの問いに順に答える：

1. **その図は文字を読ませるか？** — はい → 候補は HTML格納器 か 合成。いいえ → SVG幾何要素
2. **その図は変形・派生計算・path morph・連続的な形態変化を含むか？** — はい → 候補は SVG幾何要素 か 合成。いいえ → 文字を読ませるなら HTML格納器
3. **両方必要なら 合成**。外枠を SVG で持ち、文字を持つサブパーツだけ `<foreignObject>` でくり抜く

迷ったときの安全側は **HTML格納器**（文字を含むなら）。SVG の `<text>` で書くと文字幅を実測しないとはみ出る——その不確実性を一段消せる。

## 3. 図の設計の原則

**背骨を1本通す。** 動画全体を、ひとつの中心の図が姿を変えていく一本の物語として組む。ほかの図は背骨を順に加工していく工程として連なる。背骨があると、視聴者は画面が切り替わっても「同じ話の続き」として追える。

**ひとつの図はひとつの仕組みを見せる。** 「この図は何を見せるか」を一言で言えるかを基準にする。一言で言えない図は、たいてい2つの役を兼ねていて、どちらも届かない。

## 4. セクション別デザイン

`script.md` の1画面 = 出力の1サブセクション。各画面で次の4項目を埋める。

### 眼目

絵だけで伝える中心概念（1文）。この画面が動画から消えたときに、欠落するもの。

### 舞台 — 最も混む瞬間の止め絵

その画面が会話のあいだに通る複数状態のうち、**最も混む瞬間の画面の絵**を、読み手が頭の中で描けるレベルで言葉にする。

**配置はレイアウト構造で書く**。「左半分・右半分」「左から ○ px」のような絶対指定ではなく、ネストした構造で：

```
画面 = VStack(gap=大) [
  シーンタイトル,
  HStack(gap=中, align=center) [ Chip, FlowArrow, MassGrid ],
  字幕帯余白
]

MassGrid = Grid(24列×16行, gap=極小) of MassCell

防御カード = VStack(gap=小) [
  見出し,
  各項目 = HStack(gap=小) [ ラベル, チェック ] を 4 つ
]
```

書くこと：

- 画面全体のレイアウト構造（VStack/HStack/Grid を入れ子で）
- 主役（背骨の図）はどこにいて、どの大きさで、どの状態か
- 補助の図たちはどの構造の中に置かれ、それぞれが何を見せているか
- 画面の上下左右に意味を割り当てるなら、その対応（左＝入力／右＝出力 など）
- 余白の位置（字幕帯のぶんの下端余白）

「最も混む瞬間」を基準に据えると、画面のキャパが先に決まる。それより少ない瞬間は自然とこの止め絵の部分集合になり、各図の置き場所が他の図とぶつからずに済む。**レイアウト構造で書いておけば、tsx 段で子要素が増減したときも親が自動的に並べ直す**——「カードを1枚増やしただけで他のカードの座標を全部書き直す」事故を防げる。

カードをまたぐ矢印など、レイアウト構造の枠を越えて要素を繋ぐ場合は「**オーバーレイ層に Arrow(from=Chip, to=MassGrid)**」のように、繋ぐ相手を構造内のオブジェクト名で参照する（座標は tsx で派生計算）。

### 時間順の状態遷移

最も混む瞬間に至るまでの動きを、**ステップ列**で書く。各ステップで「何が」「どう変わる」を動詞で書き、関連するセリフ近辺を添える（spec 段でこれを event に翻訳する）。

書式：

```
(a) 〜のセリフあたり：軸が左から右へ描き起こされる
(b) 〜のセリフあたり：軸の左端に完全砂嵐パッチが、ふわっと浮かび上がる
(c) 〜のセリフあたり：軸の上に一手アイコン（DIFFUSION 色の小さな三角）が現れる
(d) 一拍おいて：(c) の右隣に、ノイズが少し薄い絵が地続きに現れる
(e) (c)(d) を 6 回くり返す。各回は次第にテンポを上げる
(f) 最後のセリフあたり：右端に猫の線画がはっきり立つ。直前にひと呼吸の間
```

書きながら確かめたい3点：

**動きの描写は、見えるさまで書く。** 「現れる」だけだとフェードイン以外の絵が浮かばない。「ふわっと浮かび上がる」「左から滑ってきて止まる」「主役の輪郭をなぞって描き起こされる」のように、画面で実際に起きる動きを動詞で言うと、tsx 段で何を実装すればいいかが残る。

**登場と退場には、出生元と退場先がある。** 図が「画面外から湧く」「フェードして消える」だけで処理されると、視聴者は要素同士のつながりを失う。「主役の右肩から枝分かれする」「補助の枠の中に吸い込まれて消える」「親 HStack の右隣セルに滑り込む」のように、既存の図やレイアウト構造に紐付くと、動画全体が地続きの一場面に見える。

**作り直すより、姿を変える。** 既存の図を消して新しい図を出すのではなく、連続変形（モーフ）で姿を変える。「{図} が {状態A} から {状態B} へ連続変形」と明示すると、tsx 段で別物として再実装される事故が減る。

### 演出ノート

止め絵と遷移を補強する、表情の指示。書くこと：

- **テンポと強弱** — ゆっくり開く／さっと閉じる／ためらいの一拍を入れる／クライマックスでひと呼吸止める
- **カメラワーク**（あれば） — ズームイン／引き／パン。何を凝視させ、何を俯瞰させるか
- **象徴的なディテール** — 完成した瞬間に外枠だけが光る／失敗の矢が落ちると地面が少し沈む／最後の一手が決まるとき、ほかの要素が一段薄くなる
- **避けたい誘惑**（あれば） — ループする揺れ、複数要素の同時アニメ、装飾としての回転——本筋を支えるための具体的な抑制を書いておくと、tsx 段で誘惑に負けにくい

## 1画面が破綻しない感覚

詰め込み・同時並行・対比過多・重なりが、視聴者の理解を取りこぼす4大要因。書きながら止まって見直す手がかり。

**画面が静かに見えるか。** 補助の図が4つも5つも並んでいると、視線がどこに着地すべきか分からなくなる。背骨・字幕・セクションタイトル以外が3つ前後に収まっていると、画面が呼吸する。超えたくなったら、本筋から外れた図が紛れているか、画面を割ったほうがいい合図。

**動きが一本道か。** 視聴者の視線は一点しか追えない。別の図が同時に動き出すと、どちらも届かない。因果的に一体の動き（視点移動＋登場 など）は1拍にまとめてよいが、無関係な2つを並行で走らせると、両方が薄まる（→ `00_philosophy.md` 3）。

**対比軸が1つに絞れているか。** 「2方式を比べる」「得手・不得手を見せる」「ゴールが同じだと示す」を1画面に詰めると、視聴者はどの軸で見ればいいか迷う。詰めたくなったら画面を割る——足し算で見せ場を増やすほど焦点が逃げる。

**最も混む瞬間が一枚のレイアウト構造に収まっているか。** トップレベルが2層以上の VStack/HStack のネストで書け、各オブジェクトがいずれかのセルに居場所を持っていれば、座標衝突は原理的に起きない。「ここはオーバーレイで重ねる」が必要な要素は1〜2点に絞る——多いと結局手動配置になる。

## レイアウトの約束

- 画面下部 220px は対話字幕の帯（HTML、半透明白の帯＋話者名＋本文 2 行まで）が常駐。主役の図はこの帯を避けて上側に配置
- シーンタイトルは画面上端の固定帯
- スマホ視聴前提。1画面の要素を少なく、余白を多く
- 画面切替は標準30fクロスフェード（境界処理は一律でよい）
- **画面トップレベルは VStack/HStack/Grid のいずれかから組み始める**。トップレベルの絶対座標は座標系の中心 (0,0) と画面端の参照だけに使う
- **レイアウト構造の中に居ないオブジェクト**（カードをまたぐ矢印、画面全体に被るフラッシュ、漂う数式ラベル など）は「**オーバーレイ層**」として明示する。多用しない——舞台に並ぶ主役は全部いずれかの構造の中に置く

## 採否判断

書き終わった原稿を、こういう状態に近づいたかで見返す。

**世界観が動いているか** — 形・色・動きが、それぞれ意味を運ぶ装置として組まれている。色が途中で別概念に乗り換えていない。

**図カタログが完結しているか** — 各図の絵が、カタログを読むだけで頭に再現できる。**全図に描画区分（HTML格納器／SVG幾何要素／合成）が指定されている**。背骨が1本通っていて、各図が「何を見せるか」を一言で言える。

**各画面の止め絵が描けるか** — 「舞台」を読んで、最も混む瞬間が一枚の絵として浮かぶ。**配置がレイアウト構造（VStack/HStack/Grid の入れ子）で書かれている**——絶対座標や「左半分」のような対象不明の領域指定が紛れていない。

**動きが連なっているか** — 状態遷移の各ステップが動詞で具体的に書かれ、登場・退場が既存の図かレイアウト構造に紐付いている。各ステップに関連セリフが添えてあって、spec 段で event に落とせる。

**画面が呼吸しているか** — 補助の図が増えすぎていない。同時に動く図がない。対比軸が1つに絞れている。オーバーレイ層の要素が1〜2点に収まっている。

**数値に踏み込んでいないか** — ピクセル座標・フレーム数が紛れていない。サイズは相対、タイミングはセリフを基準にした言葉に留まっている。
