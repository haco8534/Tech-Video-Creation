# [3] セリフ単位スペック（design_spec.md）

`design_overall.md` の「背骨」「視覚の主役」「視覚語彙」「各画面の絵の眼目／代表止め絵」を入力に、各画面で**どの装置を出し、どの動きを起こすか**を決めて event に分解し、`script.md` のセリフ1行に紐付ける段。

ここで装置と動きを決める。上流（overall）は一文しか持っていないので、装置の選び方と動き方はこの段の判断。下流（tsx）に渡すまでに、すべてが table の形で固まっている状態にする。

この段で一番避けるべき失敗は、`design_overall` の名詞を SVG 部品に置き換えるだけのこと。`探索` と書いて枝を置く、`学習` と書いて円を置く、`評価` と書いてメーターを置く、では足りない。**主役オブジェクトの状態が、セリフごとにどう変わり、その変化の跡が代表止め絵にどう残るか**を契約する。

## 画面ごとの契約

各画面について、event 表の前に次を書く。

- **主役オブジェクト** — 視聴者が最初に見るもの。字幕や見出しより強く見えること
- **具体状態** — そのオブジェクトが何を持っているか。空の盤、空の箱、名前だけの装置にしない
- **主役のサイズと居場所** — 図領域のどこを占めるか。代表止め絵で主役または主役から生まれた構造が図領域の 50% 以上を占めること
- **題材固有の視覚語彙** — この画面で使う形・色・線・配置の意味
- **因果の一筆書き** — 視線がどこから始まり、どの変化をたどり、どこで止まるかを1文で書く。ここが書けない画面は、部品を並べているだけ
- **禁止する逃げ** — 箱、ラベル、説明札で代用してはいけない概念
- **代表止め絵** — その画面の最も情報量が多い瞬間。止まっていても何の仕組みかわかる一文

## event = 名前付きの動作ビート

ある装置のある挙動を、起こす単位として一つの event で表す。それぞれを `script.md` の特定のセリフ1行に紐付ける。そのセリフの開始フレームが、tsx で Track のキーフレーム位置になる。

命名は `{オブジェクト}.{動作}` の形で自由に（例：`head.in` `head.move` `qkv.split` `summary.out`）。

event は「出る」だけで終わらせない。各 event は原則として次のどれかを起こす。

- **変形** — 主役が別の状態へ連続変形する
- **分岐** — 主役から複数の可能性が伸びる
- **選別** — 一部が残り、一部が落ちる
- **集約** — 複数の入力がひとつの値・点・一手へ集まる
- **反転** — 視聴者の初期理解と違う構造が見える
- **循環** — 結果が前段へ戻って、次の状態を変える

弱い event 名の例: `box.in` `label.show` `icon.appear` `meter.in`。  
強い event 名の例: `branch.prune` `heat.merge` `loss.backprop` `policy.bias` `tokens.rebind`。

## 画面切替

各画面の先頭セリフに `scene.{画面名}.in` を必ず付ける。これが前画面のフェードアウト → 新画面のフェードインを起こす。

## 同時表示の上限

**ひとつの画面に、同時にいる装置が 3〜4 個を超えそうなら、設計をやり直す**。やり直しの選択肢は次のどれか：

- **時間順畳み** — 古い装置に「死」の event を入れて、新しい装置が立ち上がる前に畳む（画面はリセットしない）
- **画面分割** — 一文に詰まりすぎているサイン。`design_overall` まで戻って画面を割る
- **装置を削る** — 装置を 1〜2 個減らして、残りで一文を成立させる

画面に出てくる装置の総数ではなく、**同時にいる数**を眼で見る。役目を終えた装置を消さずに次を足し続けると、最後の止め絵が崩壊する。

ただし、装置数を減らすために主役を小さくしてはいけない。主役が小さくなった時点で、画面は説明図に負けている。補助装置を消す、主役上に統合する、または画面を割る。

## 抽象逃げ検査

event 表を書く前に、各画面で次を確認する。

- ラベルを全部隠しても、主役が何をされているか見えるか
- 画面内の最も大きいものが、題材固有の具体物か
- 丸、箱、汎用アイコン、メーター、矢印が主役になっていないか
- 前の event の結果が、次の event の発生源になっているか
- 横並び比較の場合、左右の差分が「本数」「ラベル」だけでなく、状態の差として見えるか

どれかが No なら、event を増やすのではなく、主役の状態設計をやり直す。

## 代表スナップ候補

各画面の契約には **代表スナップ候補** を 2〜3 個書く。これは実装後に still snapshot を撮るフレームの候補になる。フレーム数ではなく、event 名で書く。

- `scene.X.in + 少し後` — 主役が立ち上がった瞬間
- `中盤の変化 event + 少し後` — 変化が起きている瞬間
- `代表止め絵 event + 少し後` — その画面の最も情報量が多い瞬間

後続の `04` はこの候補から静止画を撮る。代表スナップ候補が書けない画面は、何を確認すれば成功なのか決まっていない。

## 出力の形

`design_spec.md` に次を書く。

- **画面ごとの契約** — 主役オブジェクト / 具体状態 / 題材固有の視覚語彙 / 禁止する逃げ / 代表止め絵
- **event 一覧**（発生順の表）— event 名 / 紐付くセリフの抜粋 / 発生源 / 動詞 / 結果状態
- **各オブジェクトの誕生と死** — どの event で opacity 0→1 になり、どの event で 1→0 になるか。**「画面退場で死ぬ」だけのオブジェクトが多すぎたら、同時表示の上限に引っかかっていないか確認する**
- **代表スナップ候補** — 画面ごとに 2〜3 個。実装後の確認点

`何が起きるか` とだけ書くと、箱やラベルの出し入れで済ませやすい。必ず `発生源`、`動詞`、`結果状態` に分ける。

例:

| event | 紐付くセリフ（抜粋） | 発生源 | 動詞 | 結果状態 |
|---|---|---|---|---|
| king.safety | 王の安全 | 盤上のキング周辺 | 逃げ道が閉じる | 閉じたマスが赤く残り、評価メーターへ流れる |
| token.cut | 小さな部品へ切る | 文の帯 | 切れ目が走って分かれる | 部品化したトークン列として並ぶ |

座標値・フレーム数は書かない。

## 最終チェック

`design_spec.md` を出す前に、画面ごとに次を1行で自己採点する。

| 画面 | 主役は図領域の50%以上か | ラベルなしで読めるか | 前後差が止め絵に残るか | 汎用図形が主役化していないか |
|---|---|---|---|---|

Yes で埋めるための表ではない。No が出たら、その画面だけ `design_overall` へ戻す。
