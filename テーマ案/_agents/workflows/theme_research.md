# 技術系解説チャンネル テーマ案リサーチワークフロー v2

---

## 🎯 このワークフローの目的・コンテキスト

### チャンネルのコンセプト

**「言われてみると、たしかに知りたい。——そして語りたくなる。」**

身近な技術やITの仕組み・歴史・構造を、**「なぜそうなっているのか？」** という切り口で掘り下げるチャンネル。
ただ「知識として面白い」だけでなく、**視聴者が自分の経験と結びつけ、意見を言いたくなる**テーマを優先する。

### バズの3層構造モデル（v2の核心）

> [!IMPORTANT]
> このワークフローは「VSCodeテーマが他テーマの50倍バズった」という実績データから導出した
> **バズの3層構造モデル** に基づいている。

```
バズポテンシャル ＝ コンテンツ品質 × アイデンティティ関与度 × 論争性

Layer 1（最強）: アイデンティティ・トリガー
  → 視聴者の「自分は何者か」に触れるテーマ
  → 自分の選択を正当化したい／反論したい心理でクリック

Layer 2（拡散エンジン）: 論争性
  → 「正解がない問い」で全員が意見を言いたくなる
  → コメント欄活性化 → アルゴリズムが推す → インプレッション爆増

Layer 3（土台）: 認知ギャップ
  → 「言われてみれば確かに、なんでだろう？」
  → Layer 1+2と掛け合わさったとき初めて爆発する
```

### 成功パターンと失敗パターン

| パターン | アイデンティティ | 論争性 | 認知ギャップ | 結果 |
|---------|----------------|--------|------------|------|
| **VSCode（50倍バズ）** | ✅ エディタ選択＝自分の信条 | ✅ Vim/Emacs/VSCode派閥 | ✅ なんでだろう | 🔥 爆発 |
| JPEG圧縮の仕組み | ❌ 誰もJPEG派を名乗らない | ❌ 議論が起きない | ✅ 面白い | 📉 低調 |
| USB統一されない | ❌ USB-A派はいない | ❌ 全員「統一しろ」で一致 | ✅ 面白い | 📉 低調 |

### このワークフローが解決すること

| 旧v1の問題 | v2の解決策 |
|-----------|-----------|
| 「品質は高いのにバズらない」テーマを量産 | **バズの3層構造モデル**でクリック・コメント・共有の動機を事前検証 |
| 5軸スコアリングに「拡散力」の軸がない | **⑥アイデンティティ関与** **⑦論争性** の2軸を追加（/35点満点） |
| 「へー」で終わるテーマが上位に来る | **派閥テスト**（最低2陣営）を通過しないと推薦入りしない |
| コメント欄の活気を設計していない | Step 3でコメント誘発ポイントを明示的に設計する |

### 出力物の最終像

```
d:\myfolder\動画生成\技術解説\テーマ案\
  └── {日付}_{バッチ名}/
        ├── theme_proposals.md      ← 最終テーマ案リスト（7軸スコアリング・優先度付き）
        ├── ideation_log.md         ← 発散・収束プロセスの記録
        └── research.md             ← 有望テーマの市場・技術・バズ可能性リサーチ結果
```

## 📋 前提・ルール

- **出力形式**: Markdownファイル（`.md`）
- **出力先**: `d:\myfolder\動画生成\技術解説\テーマ案\{日付}_{バッチ名}/` （例: `2026-03-22_バズ狙い総合`）
- **承認ゲート**: なし。全ステップ自動進行
- **テーマ案の数**: **30〜40案**を最終候補として列挙する
- **推薦テーマ数**: スコアリングで上位 **10案**を「推薦テーマ」として詳細を提示
- **品質方針**: **「見たあとに語りたくなる」**テーマを最優先。「へー面白い」で閉じるテーマは避ける
- **⚠️ NGテーマ**: 以下のようなテーマはこのチャンネルに合わないので除外する：
  - **チュートリアル型**（○○のインストール方法、○○の使い方）
  - **最新ニュース速報型**（○○がリリースされました！）
  - **全員が同意する話**（「USBは統一しろ」のように議論が生まれない一方的主張テーマ）
  - **前提知識が高すぎるもの**（非エンジニアが入口すら持てないテーマ）

## 🧭 テーマ発掘の7つの視点

テーマ候補を探す際に、以下の7つの視点を意識する。
特に **視点6・7はv2で最重要視する新視点** であり、バズのLayer 1・2に直結する。

### 視点1：実は知らない仕組み型（ブラックボックス解体型）

> 「毎日使っているのに、中身を知らない」

- 例: Deleteキーを押すとファイルはどうなるのか？ / URLを入力したとき何が起きるか？

### 視点2：なぜこうなった？型（歴史的経緯型）

> 「今の技術の形には、歴史的な理由がある」

- 例: なぜキーボードはQWERTY配列なのか？ / JavaScriptは10日で作られた

### 視点3：なぜこんなにたくさんある？型（乱立・分岐解明型）

> 「似たようなものがたくさんあるのには、それぞれの理由がある」

- 例: 画像形式がなぜたくさんあるのか / プログラミング言語はなぜ700種類以上あるのか

### 視点4：なぜみんな使ってる？型（デファクトスタンダード解明型）

> 「支配的になったのには、技術以外の理由もある」

- 例: どうしてみんなVSCodeを使ってるの？ / Dockerはなぜ流行ったのか？

### 視点5：これ、実は違うんです型（よくある混同・誤解型）

> 「似ているけど実は全然違う。使い分けには理由がある」

- 例: OSとカーネルの違い / 暗号化とハッシュ化の違い

### 視点6：🔥 あなたの○○は正しいのか？型（アイデンティティ挑発型）【v2新設・最重要】

> 「視聴者の選択・信条・経験を問いかける」

- 視聴者が **自分の立場で見る** テーマ。自分の選択を正当化したい、反論したいという心理がクリック動機になる
- **鍵**: テーマが視聴者のアイデンティティ（使っている言語、ツール、OS、キャリアパス等）に紐づいていること
- 例:
  - なぜプログラマーはMacを好むのか？
  - Pythonの天下は本当に終わるのか？
  - 「独学エンジニア」は本当に通用するのか？
  - ChatGPTにコードを書かせるのはプログラミングか？

### 視点7：🔥 答えのない論争型（コメント爆発型）【v2新設・最重要】

> 「正解がないからこそ、全員が語りたくなる」

- **最低2つ、理想的には3つ以上の「派閥」が存在する**テーマ
- コメント欄の活性化 → YouTubeアルゴリズムがエンゲージメント高と判定 → おすすめに表示 → バズ
- **鍵**: 動画内で「これが正解」とは断言せず、構造を解説した上で視聴者の判断に委ねる設計
- 例:
  - TypeScriptはJavaScriptを救ったのか、殺したのか
  - OOP vs 関数型——なぜ答えが出ないのか
  - CSSはプログラミング言語か？
  - 結局、最強のプログラミング言語は何なのか？

## Step 0 ── 入力確認 & バッチ名決定

ユーザーからの入力を確認する。以下のパターンに対応する：

### パターンA：技術カテゴリ指定あり
> 例：「OS系のテーマを探して」「プログラミング言語の仕組み系で」

→ 指定カテゴリに絞って探索。ただし**視点6・7のバズ可能性が高いテーマを優先的に発掘する**。

### パターンB：カテゴリ指定なし
> 例：「バズるテーマを探して」「次の動画を考えたい」

→ 7つの視点（特に視点6・7を重点化）から幅広く探索する。

### パターンC：シードテーマあり
> 例：「Pythonについて作りたい」

→ シードテーマを出発点に、**「このテーマでどうすれば論争が生まれるか」** を軸にした発散を行う。
**⚠️注意: シードテーマをそのまま「○○の仕組み」にしない。「○○は本当に○○なのか？」の問いかけ形に変換することを試みる。**

### バッチ名の決定

入力内容に基づき、出力フォルダのバッチ名を決定する。
- カテゴリ指定あり → `{日付}_{カテゴリ名}`
- カテゴリ指定なし → `{日付}_バズ狙い総合`
- シードテーマあり → `{日付}_{テーマキーワード}`

### 既存テーマの除外チェック

`d:\myfolder\動画生成\技術解説\台本作成\` 配下を確認し、既に作成済みのテーマをリストアップする。
これらのテーマは最終候補から除外する（ただし「続編」「別角度」は可）。

確認が完了したら自動的に Step 1 へ進む。

# ◇ Diamond 1：探索ダイヤモンド

**目的**: シードから離れ、バズポテンシャルの高いテーマ候補を大量に生み出す

---

## Step 1 ── 発散：シード分解 & 多層連想 & バズ構造スキャン

### 目的

**80〜120個の生アイデア断片**を生成する。
v1と異なり、**「議論を呼ぶか？」「視聴者のアイデンティティに触れるか？」** を常に意識しながら発散する。

### 1-1. シード分解（Seed Decomposition）

シードテーマを以下の6つの軸に沿って分解する：

| 分解軸 | 質問 | 例（シード：「プログラミング言語」） |
|--------|------|--------------------------------------|
| **技術レイヤー** | この技術はどのレイヤーに関わるか？ | 構文、コンパイラ、ランタイム、エコシステム |
| **前提・思い込み** | みんなが無意識に信じていることは？ | 「Pythonは遅い」「静的型付けが正義」 |
| **関連する技術・概念** | 隣接する技術要素は？ | フレームワーク、パッケージマネージャ、IDE |
| **歴史・発展** | この技術はどう進化してきた？ | Fortran→C→Java→Python→Rust |
| **日常接点** | 一般の人がこの技術に触れるシーンは？ | Webサイト、アプリ、ゲーム |
| **🔥 派閥・陣営** | この技術を巡って対立する立場があるか？ | 静的型 vs 動的型、OOP vs 関数型、○○言語信者 vs △△言語信者 |

### 1-2. 3層連想マップ（Three-Layer Association）

v1と同じ3段階連想を行うが、**Layer 3で「この話題で誰が何と言うか」を意識的に追加する**。

```
[シードの技術要素] ← Layer 0（出発点）
    │
    ├── [直接的な技術連想] ← Layer 1（近い連想）
    │       │
    │       └── [関連する仕組み・歴史] ← Layer 2（遠い連想）
    │               │
    │               └── [論争・対立構造・お気持ち] ← Layer 3（★バズの種）
```

**例**:
```
プログラミング言語の選択（Layer 0）
├── Python, JavaScript, Rust, Go（Layer 1）
│   ├── 型システムの設計思想、メモリ管理の哲学（Layer 2）
│   │   └── 「Pythonは遅いから本番に使うな」vs「開発速度が正義」（Layer 3 ★）
│   └── パッケージマネージャ地獄（Layer 2）
│       └── 「node_modules重すぎ問題」は全員共感→コメント欄が活気づく（Layer 3 ★）
```

### 1-3. 「語りたくなる」質問による強制発想（Buzz Trigger Generator）

v1の「言われてみると」パターンに加え、**バズの種を意識したパターン**を追加：

| パターン | 質問テンプレート | 生成されるアイデア例 |
|---------|----------------|-------------------|
| **裏側** | 「○○するとき、裏では何が起きている？」 | URL入力時の0.3秒の旅 |
| **なぜ今の形** | 「○○はなぜ今の形・仕様になっている？」 | なぜメールアドレスに@が使われるのか |
| **なぜ消えない** | 「古い技術○○はなぜまだ生き残っている？」 | FAXはなぜ2026年でも使われるのか |
| **なぜ統一されない** | 「○○はなぜ1つに統一されないのか？」 | USBケーブルはなぜ統一されない |
| **🔥 あなたの選択は正しいか** | 「○○を使い続けて本当に大丈夫？」 | Pythonの天下は本当に終わるのか？ |
| **🔥 きのこたけのこ構造** | 「○○派 vs △△派、結局どっちが正しい？」 | OOP vs 関数型、答えが出ない理由 |
| **🔥 常識への挑発** | 「『○○は当たり前』は本当か？」 | テストを書かないコードは本当にゴミか？ |
| **🔥 未来への不安** | 「○○は今後どうなる？あなたの仕事は？」 | ChatGPTにコード書かせるのはプログラミングか？ |

### 1-4. テクノロジー × バズ構造スキャン

Web検索で**技術トレンド × 論争性**の交差点にあるテーマを発掘する。
最低6回の検索を行う：

| 検索クエリのパターン | 例 |
|--------------------|-----|
| `{技術} 論争 議論 2025 2026` | 「プログラミング言語 論争 議論 2026」 |
| `{技術} vs ○○ Reddit` | 「Python vs Rust Reddit」 |
| `{技術} is dead 2026` | 「Python is dead 2026?」 |
| `{技術} 不要 オワコン OR まだ使う` | 「jQuery 不要 オワコン OR まだ使う」 |
| `why do programmers {動詞} {テーマ}` | 「why do programmers hate Java」 |
| `{技術} hot take unpopular opinion` | 「programming hot take unpopular opinion」 |

以下を `ideation_log.md` に記録する：

## Step 2 ── 収束：クラスタリング & 3層フィルター

### 目的

Step 1で生成したアイデア断片を**15〜20個のテーマクラスタ**に統合し、
バズの3層構造モデルに照らして選別する。

### 2-1. 親和図法（Affinity Mapping）

v1と同じ。ただしクラスタ名は **問いかけ形**を推奨（例: ✕「OSカーネル」→ ○「なぜLinuxを使えないとダメと言われるのか？」）

### 2-2. バズ3層フィルター

各クラスタを以下の3層で評価する。**スコアの高い順に有望候補とする**：

| 層 | チェック質問 | 配点 |
|----|------------|------|
| **Layer 3: 認知ギャップ** | 「言われてみれば」感があるか？ | 0-2 |
| **Layer 2: 論争性** | 少なくとも2つの「派閥」が存在するか？コメント欄で意見が割れるか？ | 0-3 |
| **Layer 1: アイデンティティ** | 視聴者が「自分のこと」として見るか？自分の選択・経験が関わるか？ | 0-3 |

**合計8点満点。5点以上のクラスタを有望候補とする。**

### 2-3. チャンネル適合フィルター

残ったクラスタを7つの視点のどれに該当するかタグ付けする。
**視点6（アイデンティティ挑発型）・視点7（答えのない論争型）に該当するクラスタは必ず残す。**

### ✅ Step 2 の出力

`ideation_log.md` に追記：

```markdown
# Step 2：クラスタリング & 3層フィルター

## クラスタ一覧
| # | クラスタ名 | 認知ギャップ(/2) | 論争性(/3) | アイデンティティ(/3) | 合計(/8) | 通過 |
|---|-----------|----------------|-----------|-------------------|---------|------|

## 有望クラスタ（Step 3 へ進むもの）
（10〜15個のクラスタを選定。選定理由とバズの種を明示）
```

# ◇ Diamond 2：深化ダイヤモンド

**目的**: 有望クラスタを具体的な「バズる動画コンセプト」に磨き上げる

---

## Step 3-A ── 発散：8つの思考レンズによる角度開発

### 目的

各有望クラスタに対して8つのレンズを当て、**異なる切り口・構成・タイトル案**を生み出す。
v2では**バズ特化の2レンズを新設**している。

### 8つの思考レンズ

#### レンズ1：レイヤー解体（Layer Dissection）
> 「この技術を上から下まで分解したら、各レイヤーで何が起きているか？」

#### レンズ2：設計思想の対立（Design Philosophy Clash）
> 「この技術の設計では、どんなトレードオフが存在するか？」

#### レンズ3：歴史の分岐点（Historical Turning Point）
> 「この技術の運命を決めた、歴史的な分岐点はどこか？」

#### レンズ4：身近な比喩変換（Everyday Analogy）
> 「この技術の仕組みを、日常のモノやコトに例えたら何になるか？」

#### レンズ5：敗者の技術史（The Losers' History）
> 「勝者の陰には、消えていった技術がある。なぜ負けたのか？」

#### レンズ6：「実は全員間違えてる」テスト（Common Mistake Test）
> 「この技術に関して、技術者ですら正確に説明できないことは何か？」

#### レンズ7：🔥 派閥マッピング（Faction Mapping）【v2新設】
> 「このテーマに関する主要な陣営・立場を列挙し、それぞれの主張を整理する」

- **各派閥の名前**と**代表的な主張**を3つ以上列挙
- 各派閥が「自分が正しい」と信じる根拠を明文化
- **どの派閥が最大で、どの派閥が最もコメントしたがるか**を分析
- 例（VSCodeテーマの場合）:
  - VSCode派「拡張性最高、全部入り」
  - Vim派「軽さは正義。キーバインドの効率は他では出せない」
  - JetBrains派「型推論とデバッグはIDEの専売特許」
  - Emacs派「自分だけの環境を50年磨いてきた」

#### レンズ8：🔥 挑発タイトル設計（Provocative Title Design）【v2新設】
> 「このテーマを最もクリックしたくなる問いかけに変換する」

- 「○○の仕組み」のような教科書型タイトルを禁止
- 代わりに以下のパターンでタイトルを3案以上生成:

| タイトルパターン | 例 |
|----------------|-----|

## Step 5 ── 収束：7軸スコアリング & 最終推薦

### 目的

Step 4のリサーチ結果を統合し、**7軸スコアリング**でテーマを評価して最終推薦を決定する。

### 5-1. スコアリング基準（7軸×5段階＝最大35点）

| 軸 | 評価基準 | 配点 |
|----|---------|------|
| **① 日常接点** | 技術に詳しくない人でも「知ってる」と入口を持てるか | /5 |
| **② 差別化余地** | 競合と差別化でき、独自の角度が出せるか | /5 |
| **③ 技術的深度** | 15-20分間ファクトフルに語れる仕組み・歴史・設計思想があるか | /5 |
| **④ 好奇心トリガー** | 「えっ、そうだったの？」と言わせるポイントがあるか | /5 |
| **⑤ スッキリ感** | 「なるほど！」と腹落ちする体験を提供できるか | /5 |
| **⑥ 🔥 アイデンティティ関与** | 視聴者が「自分ごと」として見るか？自分の選択・経験・立場に関わるか？ | /5 |
| **⑦ 🔥 論争性** | 「正解がない問い」か？コメント欄で意見が割れるか？複数の「派閥」が存在するか？ | /5 |

### スコアの目安

| スコア | ⑥ アイデンティティ関与 | ⑦ 論争性 |
|--------|----------------------|----------|
| 5 | 全エンジニアが当事者。自分の選択を正当化/反論するためにクリック | 3つ以上の派閥が激論中。コメント欄必ず燃える |
| 4 | 多くのエンジニアが経験あり。「自分はこう思う」と言える | 2つの派閥あり。SNSで定期的に議論される |
| 3 | 一部のエンジニアが共感。直接的な当事者意識は薄い | 論争ポイントはあるが、陣営が明確でない |
| 2 | 知識として興味深いが「自分ごと」感は弱い | 一般的に意見は一致する。議論にならない |
| 1 | 完全に「他人事」。教養として見るだけ | 論争の余地なし。「へー」で終わる |

### 5-2. 推薦テーマの選定

- **推薦理由**
- **動画タイトル案**（3つ。問いかけ形を最低2つ含める）
- **日常接点 → 驚き → 論争の流れ**
- **🔥 派閥構造と想定コメント**
- **想定される動画の構成概要**
- **リスク・注意点**

### 5-3. テーマ間の相性・シリーズ化の提案

- **シリーズ化可能なテーマ群**
- **先に出すべき動画**の順序提案
- **「バズ狙い → ファン定着」のローテーション提案**
  - バズ狙い（視点6・7）→ 基盤テーマ（視点1〜5）→ バズ狙い… のサイクル

| 問題 | 対処法 |
|------|--------|
| Step 1でアイデアが80個に届かない | 「語りたくなる」質問の🔥パターン（あなたの選択は正しいか・きのこたけのこ構造・常識への挑発・未来への不安）を追加実行する |
| Step 2で論争性の高いクラスタが少ない | テクノロジー×バズ構造スキャンを追加実施する。Reddit/Twitter/Hacker Newsの論争トピックを検索 |
| Step 3で派閥が2つに満たない | 「この技術を嫌う人はなぜ嫌うのか？」を検索して対立軸を発掘する |
| Step 4で論争の温度が低い | テーマの切り口を変える。「○○の仕組み」→「○○は本当に必要か？」に変換してみる |
| テーマがマニアックすぎる | 日常接点 × アイデンティティの掛け算で考え直す。「プログラマーなら全員が一家言持っているか？」が判定基準 |
| 「炎上」と「論争」の境界が不安 | テーマが**技術の選択・設計判断**に関する議論であればOK。**個人攻撃・企業批判・政治**に及ぶものはNG |
| 品質テーマとバズテーマのバランスが崩れる | 推薦TOP10に品質型（視点1〜5）を最低4つ含める。バズ狙いだけのチャンネルにならないようにする |
| 既存動画テーマとの重複 | `d:\myfolder\動画生成\技術解説\台本作成\` 配下を確認し、既に作成済みのテーマは除外する（ただし「続編」「別角度」は可） |
