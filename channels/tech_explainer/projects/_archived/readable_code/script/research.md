# リサーチ：「コードは読みやすく書け」は本当に正しいのか？

## 1. 「読みやすいコード」の起源と広がり

### Clean Code（Robert C. Martin, 2008）
- ソフトウェア業界で最も影響力のあるコーディング指針書の一つ
- 「関数は短く」「名前は意図を表すように」「コメントよりもコード自体を読みやすく」
- SOLID原則、単一責任原則など設計原則も提唱
- 世界中の企業のコーディング規約に影響を与えた

### リーダブルコード（Dustin Boswell & Trevor Foucher, 2011）
- 日本では特に人気の高いコード品質書籍
- 「コードは他の人が最短時間で理解できるように書く」が核心メッセージ
- 変数名、コメント、制御フロー、ネストの深さなど具体的指針

### 「コードを読む時間は書く時間の10倍」
- Robert C. Martinが著書で引用した有名なフレーズ
- 原典を辿ると正確な研究データではなく、経験則的な見積もり
- ただし「コードは書くより読む時間の方が長い」という一般的傾向自体は、多くの開発者の実感と一致する
- IDE使用率調査（JetBrains Developer Survey等）でもナビゲーション・リーディング時間が相当の割合を占める

## 2. 読みやすさの功績

### チーム開発での効果
- コードレビューの効率向上：読みやすいコードはレビュー時間を短縮
- オンボーディング：新メンバーがコードベースを理解する速度に直結
- バグ発見率：読みやすいコードはバグを発見しやすい（Buse & Weimer, 2010の研究示唆）
- 保守コスト：ソフトウェアの総コストの60-80%は保守に費やされるという業界見積もり（正確な割合は条件により変動）

### オープンソースでの重要性
- コントリビューターが理解できないコードには貢献できない
- Linuxカーネルのコーディングスタイルガイドラインも読みやすさを重視

## 3. 反論と限界

### パフォーマンスとの衝突
- Casey Muratori「Clean Code, Horrible Performance」（2023年の議論）
  - Clean Code原則に従ったポリモーフィズムベースのコードが、switch文ベースのコードより数十倍遅くなるケースを実演
  - 仮想関数テーブルのオーバーヘッド、分岐予測の失敗、キャッシュミスが原因
  - ただしMuratori自身も「すべてのコードをこう書けとは言っていない」と補足
- ゲームエンジン（Unreal Engine、Unity内部）
  - データ指向設計（DOD）はオブジェクト指向の「読みやすい」設計と対立することがある
  - ECS（Entity Component System）パターンはClean Codeの原則と相容れない部分がある
- データベースエンジン内部（PostgreSQL、SQLite）
  - クエリオプティマイザやストレージエンジンでは、マイクロ最適化が正当化される
- 暗号化ライブラリ
  - タイミング攻撃対策のため、意図的に「読みにくい」定時間処理を書く必要がある

### 「誰にとっての読みやすさ？」
- 初心者にとっての読みやすさ ≠ エキスパートにとっての読みやすさ
- 数学・科学計算コード
  - `a = b * c + d` は数式に馴染みのある人には `result = multiplier * coefficient + offset` より読みやすい
  - NumPy/SciPyのコードは短い変数名が慣習（論文の数式表記と一致させるため）
- シェーダーコード（GLSL/HLSL）
  - グラフィックスプログラマにとっては `vec3 n = normalize(cross(v1, v2))` が最も読みやすい
- 正規表現
  - 正規表現に習熟した人にとっては、正規表現1行の方が同等のif-else 20行より「読みやすい」
- HaskellやAPL
  - 関数合成やポイントフリースタイルは、そのパラダイムに慣れた人には簡潔で明快

### 過剰な抽象化の問題
- 「読みやすさのため」の過度なリファクタリング
  - 1行の処理を関数に切り出し、その関数がさらに別の関数を呼び、3-4層のインダイレクション
  - 「AbstractSingletonProxyFactoryBean」問題（Java界隈で揶揄されるパターン）
  - コードの実行フローを追うのが困難になる
- John Carmack（id Software創業者）の見解
  - 「深いコールスタックより、少し長くても1つの関数にまとまっている方が理解しやすいことがある」
  - 「インライン化は最も見過ごされている最適化の1つ」
- Dan Abramov（React開発者）「Goodbye, Clean Code」（2020年ブログ記事）
  - 同僚のコードを「きれいに」リファクタリングしたら、かえって変更しづらくなった経験
  - 「Clean codeは目的ではなく、手段」

### Linux カーネルの実用主義
- Linus Torvaldsのコーディング哲学
  - 「良いコードは明快なコードであり、必ずしもClean Codeの教科書通りのコードではない」
  - Linuxカーネルのコーディングスタイルは、過度な抽象化を避け、直接的な表現を好む
  - インデントはタブ8文字幅 → ネストが深くなるとすぐ横に広がる → 「ネストが深すぎるコードを書くな」という暗黙の制約になる

## 4. 現代の新しい視点

### AIコーディングアシスタントの登場
- GitHub Copilot、Cursor等の普及
- AIが書いたコードは人間が読む前提で書かれていない
- 一方でAIがコードの説明や解読を支援できるため、「読みやすさ」の基準が変わりうる
- ただし現時点ではまだ発展途上で、確定的なことは言えない

### Write-only code vs Read-only code
- Write-only code：書いた本人しか理解できないコード（Perl、APLでよく言われる）
- Read-only code：読みやすいが変更しづらい（過度に抽象化されたコード）
- どちらの極端も問題がある

## 5. バランスの実践知

### コンテキストに応じた判断
- ビジネスロジック → 読みやすさ最優先（変更頻度が高い）
- ホットパス（パフォーマンスクリティカル） → 性能優先、コメントで補足
- ライブラリのパブリックAPI → 使う側の読みやすさが最重要
- 一時的なスクリプト → 動けばよい

### Googleの「Code Readability」制度
- Googleでは「Readability」と呼ばれる社内認定制度がある
- 特定言語のコードレビューを行う資格で、コードの読みやすさの基準を統一する仕組み
- 読みやすさを個人の美学ではなく、チームの共通基準として制度化している例

### 「読みやすさ」の再定義
- 読みやすさ ≠ 短さ、≠ シンプルさ、≠ Clean Codeの教科書通り
- 読みやすさ = 「そのコードが何をしているか、なぜそうしているかが、対象読者に伝わること」
- 対象読者（audience）が誰かによって最適解は変わる
