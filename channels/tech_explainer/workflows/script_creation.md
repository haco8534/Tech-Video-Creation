# YouTube解説動画 台本作成ワークフロー

## 🎯 チャンネル方針

**「わかりやすく親しみやすい。でも説明で妥協しない技術解説チャンネル。」**

- **親しみやすさ**: キャラクター対話で敷居を下げる。専門用語は噛み砕く
- **妥協なし**: 仕組みや原理が腑に落ちるまで説明する
- **好奇心**: 「へぇ」で終わらず「もっと知りたい」と思わせる
- **誠実さ**: わからないことは正直に言う。断定を避け、議論を閉じない
- **立ち位置**: 語り手は「先生」ではなく「ある程度の知識があり、そのうえでリサーチした人」
- **対象**: 門外漢でも楽しめる。固有名詞や業界あるあるに依存しない
- **テーマから逸脱しない** ニッチな話になって門外漢の視聴者を置いてけぼりにしない。テーマの入口の広さに対して内容の専門性が非対称にならないようにする。

目標尺: 15〜18分（セリフ総量 **5,000〜6,000文字**）。承認ゲートなし（全ステップ自動進行）。

## 📐 執筆の全原則（Step 4 執筆中・Step 4.5 レビュー・Step 5a 推敲で参照）

原則は A〜G の7群。各 ID（A1, G3 等）はレビュー時の指摘参照に使う。

### A. 内容の正確性

- **A1** 断定にはスコープ（時代・地域・集団・文脈）または出典を添える
- **A2** 数字には母集団・調査主体・年代を添える
- **A3** 簡略化している自覚を台本内に残す
- **A4** 言語仕様・プロトコル仕様と、その上のライブラリ・エコシステム・慣習を混同しない
- **A5** 学術分野で確立された用語は定義通りに使う。一般的連想とのズレに注意
- **A6** 「〜のために作られた」型の因果は一次情報（作者の公式発言・公式ドキュメント）で裏付ける
- **A7** 主語の大きさを監視する（「開発者は」「ユーザーは」等の過大な一般化を避ける）
- **A8** 国・地域・時代・集団のスコープを動画内で一貫させる

### B. 構成の健全性

- **B1** 教科書型の時系列羅列にしない
- **B2** 清潔すぎる物語（複雑な現実を単一の因果や綺麗な対比に収束させる）にしない。本質的なトレードオフか歴史的事故かを一次情報で確認してから枠組みを決める
- **B3** 同じ事象・概念の起源／因果／動機を、動画内の複数箇所で異なる形で語らない。主因と副因を明示的に切り分ける
- **B4** コアメッセージの前提フレーミング自体が研究により崩れていないか。崩れる場合は本文で相対化する
- **B5** 終盤では新概念を出さない。冒頭の問いの回収のみ

### C. キャラクターの立ち位置

特有の語尾・一人称・相手呼称は `create-video-full.md` のキャラクター設定表を厳守。以下は話し方の**立ち位置**ルール。

- **C1** ずんだもんは「教える人」ではなく「調べてきた人」。権威の位置から話さない
- **C2** ずんだもんは自分の理解プロセスや勘違いを開示する余地を持つ
- **C3** 断定で結論を閉じない。主張・定義・評価を提示するときは「これはあくまで一つの切り口」「他の見方もある」場合があることを知っている。特に思想・定義・評価のように複数の立場が共存する話題では、単一の正解を押し付けない。腰の低さと懐の深さを持つ
- **C4** 評価語の主語を明示する（自分の主観か、一般的な評価か）
- **C5** めたんは結論で納得して終わらない。次の疑問に繋げる形で理解確認する
- **C6** めたんはパターンに気づいて繋げる・矛盾を指摘する・別角度の返しをする
- **C7** どちらも親しみやすく穏やかなやり取りを行う

### D. 初心者への寄り添い（離脱防止）

- **D1** 固有名詞（人名・製品名・論文・著作）の総量を抑える。覚える必要がない人名は役割ラベル＋年代に置換
- **D2** 人名を出すなら、名前を覚えなくても話の筋が通る書き方にする
- **D3** エンジニアリング特有の日常（コードレビュー・リポジトリ・リファクタリング・PR・テストコード等）を前提知識として持ち出さない。日常生活の類似構造で代替するか、最小限の導入を挟む
- **D4** 専門用語は初出時に噛み砕き、非専門家が日常で触れる類似構造と1度接続する
- **D5** 事実・統計・研究結果の直後に、視聴者の経験に接続する具体例や情景を置く
- **D6** 固有名詞の表記: 企業名・製品名・サービス名・技術規格・フレームワーク・言語名・ライブラリは**原語（英語・数字・記号）で表記**。カタカナ転写はしない。
  - ✅ `YouTube` / `Google` / `Netflix` / `Microsoft` / `Amazon` / `Apple` / `iPhone` / `H.264` / `HEVC` / `AV1` / `VVC` / `FFmpeg` / `ProRes` / `DNxHD` / `Chrome` / `Firefox` / `Zoom`
  - ❌ `ユーチューブ` / `グーグル` / `エイチ264` / `ヴィヴィ` / `プロレス` / `ズーム`
  - 例外: 慣用的にカナ表記で定着した**人名**（チューリング／ダイクストラ／ブルックス等）と、原語で書くと TTS の読み上げが著しく崩れる語（その場合はカナ+原語併記ではなく、読みやすい方を選ぶ）

### E. 面白さとテンポ（視聴体験の強度）

- **E1** 権威と数値だけで「すごさ」を作らない
- **E2** 抽象的な主張の直後に、視聴者の頭で絵が動く具体シーンを置く
- **E3** 引用を並べて論を積み上げない。引用は論拠の一部で論そのものではない
- **E4** 「えっ」と声が出る瞬間（反直感／身近な現象と本質の接続）を動画全体で複数回設計
- **E5** 最もパワーのあるフレーズや概念を一瞬で通過させない。そこで立ち止まる時間を作る
- **E6** テンポを意識する。同じ話題に長く留まりすぎない。短い応酬と長い展開を混ぜる

### F. AI的文章・構造の排除

- **F1** 数値・事実の直後に内容を考えない反射的な驚きリアクションを置かない
- **F2** 身体感覚・生理反応を表す語を、根拠なき感嘆の装飾に使わない
- **F3** 議論を一方向に閉じる形の理解確認を定型化しない
- **F4** 「事実→感想→要約」の同じ構文を連続させない
- **F5** 主張の内容ではなく強度だけを上げる形容を根拠なく重ねない
- **F6** リアクションの温度を均一化しない
- **F7** 全ブロックの長さ・リズム・往復回数を均質化しない

### G. 視聴維持率の施策

**位置別の強度配置**

| 位置 | 要件 |
|---|---|
| 冒頭15秒 | サムネ・タイトルの約束を即座に果たす。視聴者が「これを見に来た」に応える |
| 冒頭30〜60秒 | 共感フック／解消されない疑問の提示／**動画全体の見取り図シーン（G9）**。長い挨拶禁止 |
| 30〜50% 地点 | 動画で最も意外性のある事実・ピークを配置（中だるみ対策） |
| 各ブロック末尾 | 次ブロックの核を具体的に匂わせる。抽象的な予告は不可 |
| 終盤 | 冒頭の問いの明示的回収 |

**技法**

- **G1** オープンループ: 解消されない疑問を複数同時に走らせる。1つ解決する前に次を提示
- **G2** パターン中断: 視聴者の予測を裏切る変化（話題の急転換・意外な例え・視覚の切替）を定期的に挟む
- **G3** 先回り: 視聴者が抱きそうな反論・疑問を、視聴者が口にする前に台本で取り上げる
- **G4** 結論の焦らし: コアメッセージを早々に全開示しない。部分開示→展開→完全開示の段階を設計
- **G5** 共感ポイント: 視聴者が内心で頷く瞬間を散らす。業界人あるあるではなく、誰でも分かる体験
- **G6** 退屈検知: 同じ話題・同じ構文・同じトーンが続く箇所を特定し、変化を入れる
- **G7** ブロック冒頭の引き込み: ブロック開始直後の数十秒でそのブロックの核心価値を示す（長い前振りを避ける）
- **G8** 段階的情報開示: 一度に全部伝えない。「実はもう一つ」「ここからが本当の話」で深掘り
- **G9** 見取り図: 導入ブロックに「今日の全体像」を明示するシーンを独立で置く。扱う論点・順路を具体的に列挙し（「まず〇〇、次に〇〇、最後に〇〇」）、視聴者が自分の疑問がどの順で解消されるかを把握できるようにする。抽象的な予告（「深く掘っていきます」等）だけでは不可

ユーザー指定のテーマを起点に、マルチ視点で調べる。コアメッセージはここでは決めず、リサーチ結果を踏まえて Step 2 で決める。

**1a・1b・1c は同時並列で実行する（逐次実行禁止）。**
各サブステップ最低5回検索（同クエリ繰り返し禁止）、全体15件以上のファクト収集。

| サブ | 視点 | 出力 |
|---|---|---|
| **1a** | 公式データ・基本定義・代表研究・なぜ広く使われているか | `research_1a.md` |
| **1b** | 通説への反証・例外・限界。特に：通説と一次情報のズレ／因果が歴史的偶然だった例／言語仕様とエコシステムの混同／学術語と一般語の意味ズレ | `research_1b.md` |
| **1c** | 歴史的経緯・設計思想・専門家コンセンサス・日常生活への接続 | `research_1c.md` |

**1d — 統合**: 重複排除 → 信頼度再評価 → 欠落補完 → **主因・副因の切り分け**（同じ事象に複数の理由があれば一次情報で主因を確定。確定不能なら「諸説あり」扱い）→ 優先度付け → `research.md` 出力

## Step 2 — テーマ設定 & 核心メッセージ

Step 1 のリサーチ結果（特に `research.md` と `research_1b.md`）を踏まえて以下を決める。リサーチで想定していた切り口が成立しないと判明したら、ここで方向転換してよい（無理にテーマ名どおりに進めない）。

1. **コアメッセージ**（1文）
2. **ターゲット視聴者像**（プログラミング未経験者を含む想定）
3. **視聴者がよく持つ誤解・前提知識**（リサーチで判明した通説とのズレを反映）
4. **フック候補3案**
5. **視聴前→視聴後の認識変化**
6. **コアメッセージへの反論候補**（`research_1b.md` の異議を反映、3件以上）

## Step 3 — 構成設計

### 3-1. 構成方針
題材ごとに設計する。定型の型に流し込まない。決めるのは：
- 冒頭で起点にする視聴者の感情・疑問
- 視聴者のモデルをどこで・どう更新するか
- 何を最後に回収するか

### 3-2. 満たすべき性質
- 冒頭30秒以内に視聴者の注意を掴む
- 本編に少なくとも1回、視聴者の既存モデルを揺さぶる瞬間
- 終盤で冒頭の問いを明示的に回収
- 各ブロック末尾に次への接続

### 3-3. ブロック構成
導入1〜2 / 本論2〜4 / 終わり1 を目安に題材で増減。長さ・構造を均等化しない（題材が許す限り不均等を許容）。

### 3-4. 使える説明の道具（ノルマではなく引き出し）
思考実験／アナロジー／問いかけ／段階的な理解更新／対照／制約からの説明（「これが無かったらどうなるか」）／一次情報の直接参照。
**題材に合わないものは使わない。特定の道具を使うノルマを置かない。**

### 3-5. 論理追跡表（必須）
```
| # | 問い・予告 | 提示Block | 回収Block | 回収方法 |
```
回収Block空白の行は禁止。未回収が1件でもあれば Step 4 に進めない。

### 3-6. シーン遷移
- **切替タイミング**: 話題転換／情報形式変化／比較開始／エビデンス登場／まとめ開始／新しい数字／思考実験の結果
- **密度**: 1〜2分→2〜4シーン、2〜3分→4〜6、4〜5分→6〜10
- セリフ2〜3往復ごとに遷移。同じ背景4往復以上禁止

### 3-7. 1シーンの情報量
画面に出す情報単位は2〜3秒で把握できる量まで。箇条書き最大4行／表は3行×3列を超えたら分割／目次的シーンは項目5以上で分割。

### 4-2. 執筆中チェック
**全原則 A〜G を各セリフ執筆時に参照する**。特に **D（初心者寄り添い）・E（面白さとテンポ）・G（視聴維持率）は執筆中の発想段階で効かせる**（後から取り戻しにくい）。

| 推定時間 | セリフ文字数 |

| 1分 | 300〜350 |

### 4-4. ブロック横断チェック（Step 4.5 の前）
1. 論理追跡表の全行と台本セリフを照合。未回収0件
2. 各ブロック末尾→次ブロック冒頭の接続確認
3. 専門用語が説明なしに突然登場していないか
4. テーマ逸脱: 各ブロックを削除したらコアメッセージ理解に支障が出るか。出なければ削除または統合
5. **因果文の整合性**: 「〜のために作られた」「〜という理由で誕生した」「〜のおかげで普及した」を全抽出。同じ対象に異なる因果があれば主因を1つに統一、他は副因明示

## Step 4.5 — 多角通読レビュー

台本完成後、メインClaude が以下3つの視点で順番に通読する。視点を切り替えるたびにフォーカスをリセットし、その視点固有の問題だけを拾う。指摘は採用／却下／保留を内部判断し、採用分を台本に反映してから Step 5a へ進む。`review.md` への書き出しは任意（指摘が10件超で整理が必要な場合のみ）。

D/E/G（初心者寄り添い・面白さ・視聴維持率）は執筆中チェック（4-2）と推敲（5a）でカバーするので、ここでは扱わない。

### 視点1 — 辛口専門家（学術・現場・歴史を兼ねる）

学術用語の誤用・定義ズレ／論理・因果の不当な単純化／一般語と学術語の混同／言語仕様とライブラリ・エコシステム・慣習の混同／実装レベルで成立しない抽象化／現場実情との乖離／歴史的経緯の単純化・神話化／「〜のために作られた」の一次情報裏付け不足／時代・地域・文脈の混同。  
主な参照原則: A1〜A8, B2

### 視点2 — 整合性監査（全体を通読するエディタ）

同じ事象に動画内で異なる因果・起源・動機／冒頭前提と結論の整合／ブロック間論理接続の空振り／清潔な物語化のために現実の複雑さを切り捨て／コアメッセージの前提フレーミングの成立性（`research_1b.md` と照合）／教科書型の羅列／終盤での新概念導入。  
主な参照原則: B1〜B5

### 視点3 — 読み手監査（人間の読み手・キャラクター観点）

視聴を維持させられるほど面白いか／知的好奇心を刺激するか／反射的驚きリアクション／身体感覚語の根拠なき動員／議論を閉じる理解確認／構文反復／強度だけの形容／ブロック構造の同質性／人間なら選ばない不自然な言い回し／ずんだもんの権威的な位置取り／評価語の主語不明確／めたんが結論を閉じて終わっている箇所。  
主な参照原則: C1〜C6, F1〜F7

### 進め方の指針

- 各視点で台本全体を最低1パス通読する
- 「おおむね問題ない」という総評で済ませない。問題があれば該当箇所を引用して具体的に挙げる。本当に無ければ無いと結論してよい
- 視点を跨いで同じ箇所が引っかかったら優先度を上げる
- 採用判断: 題材の都合で意図的／指摘の根拠が弱い／他の原則と衝突する場合は却下してよい

## Step 5b — 概要欄生成

`script.md` と `research.md` をもとに `description.md` を生成。Phase D でタイムスタンプが埋まり `deliverables/概要欄.md` になる（uploader が読む）。

### 含める要素（この順）
1. **1行目**: 動画タイトル候補（uploader が参照）。タイトルとして成立する1行
2. **動画概要**（3〜5行、コアメッセージ反映）
3. **タイムスタンプ**: `{{TIMESTAMPS}}` プレースホルダー（Phase D で実時間に置換）
4. **参考文献**: 台本で実際に使用した🟢🟡の出典のみ（🔴除外、URLは確認済みのみ）
5. **チャンネル紹介定型文**:
   > 📘 このチャンネルについて
   > 一次情報と研究データに基づきながら、難しいテーマをずんだもんとめたんの対話で楽しくわかりやすく解説するチャンネルです。
6. **ハッシュタグ** 3〜5個

> ⚠️ タイムスタンプは音声生成後に `scene_durations.json` から計算するため、この段階では `{{TIMESTAMPS}}` のみ記載。
