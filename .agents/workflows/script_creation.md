# YouTube解説動画 台本作成ワークフロー

## 🎯 チャンネルの方向性

**「わかりやすく親しみやすい。でも説明では妥協しない技術解説チャンネル。」**

- **親しみやすさ**: キャラクター対話で敷居を下げる。専門用語は必ず噛み砕く
- **説明の妥協なし**: 「なんとなくわかった気になる」ではなく、仕組みや原理が腑に落ちるレベルまで説明する
- **知的好奇心**: 「へぇ」で終わらず「もっと知りたい」と思わせる。**なぜそうなるのか**まで踏み込む
- **誠実さ**: わかっていないことは正直に言う。過度な断定を避ける

承認ゲートなし（全ステップ自動進行）。目標尺: 約20分（セリフ総量 **6,000〜7,000文字**）。

---

## 📐 3Blue1Brown的 説明アプローチ（最重要）

事実やデータを時系列で並べるだけの「教科書型」説明は**禁止**。以下の原則で概念を構築する。

### 9つの原則

**原則1: 直感ファースト** — まず「腹落ちする直感的イメージ」を作ってから正確な説明に進む。
- ❌「マシュマロ実験は1960年代にスタンフォード大学で行われた実験です」
- ✅「もしあなたが4歳で、目の前に大好きなお菓子があって、15分触らなければもう1個もらえると言われたら？」

**原則2: 思考実験** — 「もし〇〇だったら？」を考えさせ、極端な仮定で本質を炙り出す。
- ✅「もし活性化関数がなかったら？何層重ねても直線しか引けないのだ」

**原則3: 拡張アナロジー** — 1行で終わるたとえ話ではなく、アナロジーの世界の中に留まって概念を段階的に体験させる。導入→展開A→展開B→限界を示す→本物に戻る。

**原則4: 段階的洞察（最重要構成パターン）** — ① 80%正しい簡単な理解を構築 → ② それでは説明できない現象で直感を壊す → ③ 本当の理解に導く。動画に必ず1回以上このサイクルを含める。

**原則5: パターンの発見** — 一見無関係な2つの現象が同じ構造だと気づかせる「統合の瞬間」を設計する。

**原則6: 能動的な問いかけ** — 事実を教える前に問いを投げる。めたんが予想を立て、正しい場合も間違う場合もある。

**原則7: 制約から理解する** — 「これは〇〇です」ではなく「もしこれがなかったらどうなるか」で説明する。

**原則8: ビジュアルの連続性** — シーン間で視覚要素が残り、変形・拡張・分解されて新概念に発展する。

**原則9: ビジュアルが説明そのもの** — 図解はセリフの「挿絵」ではない。ビジュアルの変化そのものが概念の説明。

> ⚠️ **最重要禁止事項**: ファクトを時系列で羅列する「教科書型」説明は絶対にしない。研究の紹介も「驚くべき事実」から入り、「なぜ？」と思わせてから詳細を出す。

## 📋 前提ルール

- **キャラクター設定**: `create-video-full.md` のキャラクター設定表を厳守
- **情報の信頼度**: 🟢高（公式・査読済み論文）、🟡中（信頼メディア・専門家発言）、🔴低（未確認→**台本使用不可**）

## Step 2 — マルチ視点リサーチ（3並列 + 統合）

**Step 2a・2b・2c は同時並列で実行する（逐次実行禁止）。**

### リサーチの量・深さ基準
- 各サブステップで**最低5回以上のWeb検索**（同じクエリの繰り返し禁止）
- 全体で**最低15件以上のファクト**収集
- 検索クエリパターン: テーマ+研究/統計/批判/歴史/誤解/具体例/専門家名/最新動向/英語

### Step 2a — 基本情報・仕組み・通説の収集
公式データ・統計・基本定義・代表的研究。テーマの「なぜ広く使われているか」の理由まで。
→ `research_2a.md` 出力（ファクト表: #/ファクト/信頼度/出典 + まとめ）

### Step 2b — 対立軸・比較・誤解されやすい点の収集
反証・例外・限界・対立視点・トレードオフ・類似概念との違い。
→ `research_2b.md` 出力（同形式）

### Step 2c — 背景・深掘り情報の収集
歴史的経緯・設計思想・専門家コンセンサス・日常生活への影響。
→ `research_2c.md` 出力（同形式）

### Step 2d — 統合（全チェックポイント完了後）
重複排除→信頼度再評価→欠落補完→優先度付け→ `research.md` 出力

**完了条件**: 🟢高 8件以上、🟡中 5件以上、3方向すべてカバー

## Step 3 — 構成設計

### 構成設計の必須チェック
- [ ] 冒頭30秒以内にフック
- [ ] 各ブロック末尾に「引き」
- [ ] 最も意外性のあるファクトは中盤（Block 3〜4）
- [ ] 冒頭の問いがまとめで回収
- [ ] 「直感の構築→破壊→再構築」サイクルが最低1回
- [ ] 「パターン統合ポイント」が最低1回
- [ ] 各トピックに核アナロジー
- [ ] 教科書型構成になっていない

### 3Blue1Brown的 構成パターン

**パターンA: 直感の構築→破壊→再構築**（動画核心で必須）
1. 80%正しい直感を丁寧に構築 → 2. それでは説明できない現象を突きつける → 3. 本当の理解に導く

**パターンB: 思考実験ドリブン**
1. 「もし〇〇がなかったら？」→ 2. その世界の問題を具体的に → 3. 「だからこそ〇〇が必要」

**パターンC: パターン統合**
1. 概念A → 2. 概念B → 3. 「実はAとBは同じ原理だった！」

### 標準ブロック構成

| Block | セクション | 内容 | 推定時間 | 3Blue1Brown要素 |
|-------|----------|------|----------|----------------|
| 1 | 導入 | フック：思考実験・常識を覆す問い | 1〜2分 | 問いで始める |
| 2 | 導入 | 前提構築：アナロジー導入 | 2〜3分 | 核アナロジー |
| 3 | 本論① | 直感的理解の段階的構築 | 4〜5分 | 直感ファースト |
| 4 | 本論② | 直感の破壊→再構築 | 4〜5分 | 段階的洞察 |
| 5 | 本論③ | パターン統合・設計思想 | 3〜4分 | パターン統合 |
| 6 | 終わりに | 冒頭の問い回収・認識変化 | 2〜3分 | 直感の対比 |

### 論理追跡表（必須）

台本中の「問い」「仮説」「予告」をすべて登録し、回収先を設計する。

```
| # | 問い・予告 | 提示Block | 回収Block | 回収方法 |
|---|-----------|-----------|-----------|----------|
| 1 | 冒頭フック | Block 1 | Block 6 | まとめで引用 |
```

**ルール**: 回収Blockが空白の行は禁止。未回収の問いが1件でもあればStep 5に進めない。

### シーン遷移設計

各ブロックの構成案にシーン遷移ポイントを含める。

**シーン切替タイミング**: 話題転換/情報形式変化/比較開始/エビデンス登場/まとめ開始/新しい数字/思考実験の結果

**1ブロックあたりのシーン数**: 1〜2分→2〜4、2〜3分→4〜6、4〜5分→6〜10。セリフ2〜3往復ごとにシーン遷移。同じ背景4往復以上は禁止。

## Step 4 — セクション別台本執筆

1ブロックずつ順番に執筆。全ブロック完了後にブロック横断チェック。

### 台本フォーマット

```markdown
---
# ▶ 導入

## 【Block 1：タイトル】
<!-- 推定時間：〇分 / 目標文字数：〇〇〇〇文字 -->

<!-- SCENE: タイトルカード | オープニング -->

めたん：〇〇〇〇
ずんだもん：〇〇〇〇

<!-- SCENE: 数値インパクト | 98.9%のWebサイト -->

ずんだもん：〇〇〇〇
...

---
# ▶ 本論

## 【Block 3：トピック①「〇〇について」】
...

---
# ▶ 終わりに

## 【Block 6：まとめ】
...
```

### キャラクターの話し方

**ずんだもん**（解説役 — 3Blue1Brown的語り手）
- 語尾「〜のだ」「〜なのだ」
- 事実を提示する前にまず問いを投げる
- 拡張アナロジーで概念を体験させる（1行で終わらせない）
- 思考実験を多用（「もし〇〇がなかったら？」「逆に無限に増やしたら？」）
- 80%の理解を構築してから壊す
- 「実はさっきの話と同じ構造なのだ」でパターン統合
- 知的興奮を共有する話し方（「ここがすごく面白いところなのだ」）

**めたん**（聞き役 — 視聴者の思考を代弁）
- 語尾「〜ですわ」「〜ですの」
- ずんだもんが答えを言う前に予想を立てる（正しい場合も、間違う場合もある）
- パターンに気づくリアクション（「これってさっきの〇〇と同じですの！？」）
- 鋭い質問で議論を深める（「それって〇〇と矛盾しませんの？」）
- 理解の確認（「つまり〇〇ってことですわよね？」）

### キャラクターの人間味（重要）

AI的な「数値復唱→驚き→きれいな要約」の繰り返しを避ける。

**ずんだもん**: テーマへの個人的感情（「鳥肌立ったのだ」）/ 自分も間違えた経験 / 好み・こだわり / 言い淀みや自己ツッコミ

**めたん**: 自分の体験（「私もそれやったことありますわ」）/ 不完全なリアクション（「えっと…あ、わかった！」）/ 個人的好み / 別角度の返し（「裏を返すと怖くないですの？」）

| ❌ AI的 | ✅ 人間的 |
|---------|----------|
| 「76%！？すごい！」 | 「76%…確かにVSCode以外使ってる人思い浮かばないですわ」 |
| 「なるほど、つまり〇〇ですわね」 | 「あー…だからあのとき〇〇だったんですわね」 |
| （常に解説モード） |  |

頻度: 1動画あたり4〜6回。各ブロック0〜1回。直後に解説に戻る。

### 文字数目安

| 推定時間 | セリフ文字数 |
|----------|------------|
| 1分 | 300〜350文字 |
| 2分 | 600〜700文字 |
| 3分 | 900〜1,050文字 |
| 4分 | 1,200〜1,400文字 |

### ブロック横断チェック（全ブロック完了後）
1. 論理追跡表の全行について台本セリフと照合。未回収0件を確認
2. 各ブロック末尾→次ブロック冒頭の接続確認（空振り・話題飛躍の検出）
3. 専門用語の初出チェック（説明なしに突然登場する用語がないか）

## Step 5a — 推敲・最終チェック

### チェック項目
1. **整合性**: 因果関係/主張の一貫性/前提の暗黙化/具体例の整合
2. **隣接セリフ論理**: 数値→感想の間に「なぜすごいか」の根拠が明示されているか
   - ❌「76%が使っている」→「50年の歴史はすごい！」（論理不接続）
   - ✅「76%が使っている」→「4人に3人が同じツールは異例」→「なぜ？」
3. **テーマ名バイアス**: テーマ名に「VS」→不要な対立構造？ / 「なぜ」→教科書型？ / 固有名詞→過度な美化？
4. **ターゲット一貫性**: Step 1の視聴者像に対して情報が適切か
5. **ファクト確認**: research.md と照合。事実誤認がないか
6. **尺**: セリフ総文字数 6,000〜7,000文字
7. **キャラクター一貫性**: 各ブロックで話し方が統一されているか
8. **セクション構造**: 「導入」「本論」「終わりに」見出し、セクション間に `---`
9. **視聴維持率**: 冒頭フック/ブロック末尾の引き/中盤の意外性/問いの回収
10. **シーン遷移マーカー**: 各ブロック冒頭にマーカー / パターン名がカタログ内 / 1シーン最大3往復 / シーン数が適切
11. **3Blue1Brownスタイル**: 核アナロジー / 問いかけ3つ以上 / 思考実験1つ以上 / 直感の構築→破壊→再構築 / パターン統合 / 教科書型なし
12. **論理追跡表**: 未回収の問い0件 / ブロック末尾の引きが次で着手 / 煽り空振りなし
13. **結論**: コアメッセージと整合 / 新概念の唐突導入なし / 最初の直感と最後の理解の対比が明示的

→ `script.md` 出力

## Step 5b — YouTube概要欄テキスト生成

`script.md` と `research.md` をもとに `description.md` を生成する。

### 含める要素
1. **動画概要**（3〜5行）: コアメッセージを反映
2. **タイムスタンプ**: ブロック構成に基づく `00:00 セクション名` 形式
3. **参考文献**: 台本で実際に使用した🟢🟡の出典のみ（🔴は除外、URLは確認済みのもののみ）
4. **チャンネル紹介定型文**:
   > 📘 このチャンネルについて
   > 一次情報と研究データに基づきながら、難しいテーマをずんだもんとめたんの対話で楽しくわかりやすく解説するチャンネルです。
5. **ハッシュタグ** 3〜5個
