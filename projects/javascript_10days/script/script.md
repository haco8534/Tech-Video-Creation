# ▶ 本論

## 【Block 3：トピック①「10日の"欠陥品"コレクション」】
<!-- 推定時間：4分 / 目標文字数：1200〜1400文字 -->

<!-- SCENE: コードビジュアル | typeof null -->

ずんだもん：ここからが楽しいのだ。JavaScriptの「おかしな挙動」を見ていこう。
ずんだもん：まずこれ。typeof nullと入力すると、"object"が返ってくるのだ。
めたん：nullって「何もない」って意味ですわよね？ なんでオブジェクト扱いに？
ずんだもん：これはガチのバグなのだ。
ずんだもん：初期実装で、nullとオブジェクトが同じ型タグ──000を共有していたのだ。
ずんだもん：つまりnullかオブジェクトか区別できない設計になっていたのだ。
めたん：10日だとそういう細かいところまで手が回らないですわよね……。
ずんだもん：でもここが面白いところなのだ。このバグは30年間直されていないのだ。
めたん：え、直せばいいのに。なんでですの？
ずんだもん：それはあとで話すのだ。ここでは「なぜ直さないのか」を覚えておいてほしいのだ。

<!-- SCENE: コードビジュアル | NaN === NaN -->

ずんだもん：次。NaN === NaN。これ、何が返ると思うのだ？
めたん：NaNはNaNと同じだから……trueですの？
ずんだもん：falseなのだ。
めたん：え！？ 自分自身と等しくないって、哲学の問題みたいですわ。
ずんだもん：これはIEEE 754っていう浮動小数点の国際規格に従った結果なのだ。
ずんだもん：NaNは「計算できない値」を意味するから、比較自体が意味をなさないっていう設計なのだ。
めたん：あ、これはバグじゃなくて仕様なんですの？ ややこしいですわ……。
ずんだもん：typeof nullはバグ、NaN === NaNは仕様。この区別が大事なのだ。

<!-- SCENE: コードビジュアル | 型強制の罠 -->

ずんだもん：そしてこれがずんだもん的に一番好きなやつなのだ。
ずんだもん：1 + "2" の結果は "12" なのだ。文字列結合になるのだ。
ずんだもん：でも "5" - 1 は 4 になるのだ。引き算だと数値になるのだ。
めたん：えっ……足し算と引き算で型変換のルールが違うんですの？
ずんだもん：そうなのだ。+は文字列結合にも使えるから、文字列が優先されるのだ。
ずんだもん：でも-は数値演算にしか使えないから、自動的に数値に変換するのだ。
めたん：一貫性がないですわね……。私もこれで3時間溶かしたことありますわ。

<!-- SCENE: コードビジュアル | ==の罠 -->

ずんだもん：もう1つだけ見せたいのだ。==（イコール2つ）の型強制なのだ。
ずんだもん：0 == false はtrue。空文字列 == 0 もtrue。でも空文字列 == false もtrueなのだ。
めたん：全部trueなんですの？ もう何が何だか……。
ずんだもん：だから今のJSでは===（イコール3つ）を使うのが常識なのだ。
ずんだもん：===は型変換をしないから、正確に比較できるのだ。
めたん：最初から===だけにしておけばよかったのに……。
ずんだもん：10日だったからね。

<!-- SCENE: テキスト強調 | でも── -->

ずんだもん：さて、ここまで聞くと「こんなの使えないじゃん」って思うよね。
めたん：正直、思いますわ。欠陥だらけじゃないですの。
ずんだもん：でもね、ここからが本当の話なのだ。
ずんだもん：この「欠陥品」が、98.9%のWebサイトに使われて、13年連続利用率1位なのだ。
めたん：……確かに、おかしいですわね。欠陥品なのに天下を取ってるって。
ずんだもん：なぜか。その答えは「品質」以外のところにあるのだ。

## 【Block 5：トピック③「壊さずに進化する──Don't Break the Web」】
<!-- 推定時間：4分 / 目標文字数：1200〜1400文字 -->

<!-- SCENE: アナロジー | 壊れないリフォーム -->

ずんだもん：さて、ここからがずんだもんの一番の推しポイントなのだ。
ずんだもん：JavaScriptの本当のすごさは「壊さずに進化する」哲学にあるのだ。
めたん：壊さずに進化？
ずんだもん：例えるとね、古い家のリフォームなのだ。
ずんだもん：壁を壊して立て直すのが普通のリフォームだよね。
ずんだもん：でもJavaScriptは「壁は一切壊さずに、部屋を増築していく」のだ。
めたん：えっ、それだと最終的にものすごく変な間取りになりません？
ずんだもん：なるのだ。でもその代わり、住んでいる人は一度も引っ越ししなくていいのだ。

<!-- SCENE: テキスト強調 | Don't Break the Web -->

ずんだもん：JavaScriptの進化を管理しているTC39っていう委員会には、鉄の掟があるのだ。
ずんだもん：「Don't Break the Web」──Webを壊すな、なのだ。
ずんだもん：新しい機能は追加してもいい。でも既存のコードが動かなくなる変更は絶対に禁止なのだ。
めたん：だから1995年に書いたJavaScriptが、2025年のブラウザでも動くんですの？
ずんだもん：そう。30年の後方互換性。これ、多くの言語では実現できていないのだ。
ずんだもん：Python 2から3への移行がどれだけ大変だったか、知っている人は分かると思うのだ。
めたん：Pythonは結局2を切り捨てましたわよね。それでみんな大変だったって聞きますわ。
ずんだもん：JavaScriptはそれを絶対にやらないのだ。

<!-- SCENE: タイムライン | ES4の悲劇 -->

ずんだもん：ところで「じゃあ最初からちゃんと作り直せばいいのに」って思わないのだ？
めたん：思いますわ。バグも直せばいいのに、って。
ずんだもん：実は一度それを試みたことがあるのだ。ES4っていうバージョンなのだ。
ずんだもん：クラスベース継承、型アノテーション、名前空間……大幅な刷新を計画したのだ。
ずんだもん：約10年かけて開発したのだ。
めたん：10年！？ それだけかけたなら相当良いものができたんじゃ……。
ずんだもん：2008年に破棄されたのだ。
めたん：えっ！？

<!-- SCENE: テキスト強調 | 大きすぎた変更 -->

ずんだもん：MicrosoftやJavaScriptの重要人物ダグラス・クロックフォードが反対したのだ。
ずんだもん：「変更が大きすぎて、既存のWebサイトが壊れる」っていう理由なのだ。
めたん：……あ。つまり「良い言語にしようとしたら、Webを壊しそうになった」んですの？
ずんだもん：その通りなのだ。ここがこの話の核心なのだ。
ずんだもん：「良い言語が勝つのではなく、壊れない言語が生き残る」のだ。
めたん：じゃあES4が失敗した後、JavaScriptはどうやって進化したんですの？
ずんだもん：2009年にES5が出たのだ。小さな改善だけ──strict modeとJSON.parseの追加なのだ。
ずんだもん：「壊さず、ちょっとずつ良くする」路線がここで確立されたのだ。
ずんだもん：そして2015年、ES6──正式名ES2015が来たのだ。
ずんだもん：let/const、アロー関数、クラス構文、Promise、モジュール……JS史上最大のアップデートなのだ。
めたん：でもそれ、かなり大きな変更ですわよね？ ES4と何が違うんですの？
ずんだもん：全部「追加」なのだ。古い書き方も全部動く。壊していないのだ。
めたん：あー、だからvarも残ってるんですわね。letを追加しても、varを消さない。
ずんだもん：そう。部屋を増築しても、古い部屋は壊さないのだ。

<!-- SCENE: テキスト強調 | SmooshGate -->

ずんだもん：「壊すな」の徹底ぶりを示すエピソードがもう1つあるのだ。
ずんだもん：2018年のSmooshGate事件なのだ。
ずんだもん：配列のメソッドにflattenって名前を付けようとしたら、古いライブラリと名前が衝突したのだ。
ずんだもん：TC39は名前をflatに変更して回避したのだ。
めたん：メソッド名1つ変えるだけでも「壊すな」を貫くんですの……徹底してますわね。
