## 1. 「プログラミング言語」の定義

### チューリング完全性

- **チューリング完全**とは、チューリングマシンと同等の計算能力を持つこと。任意の計算可能関数を表現できる
- チューリング完全な言語の条件（実用的な判定基準）：
  - 条件分岐（if/else）
  - 任意の繰り返し（無限ループが可能）
  - 任意のメモリ読み書き
- 多くの「プログラミング言語」はチューリング完全だが、**チューリング完全性はプログラミング言語の必要条件ではない**
  - SQLの基本形はチューリング完全ではないが、プログラミング言語として広く認知されている
  - 正規表現もチューリング完全ではないが、パターンマッチングの「言語」である
  - 逆に、チューリング完全であってもプログラミング言語と見なされないもの（例: Magic: The Gathering のルール、Minecraft のレッドストーン回路）もある

### 宣言型 vs 命令型

- **命令型（Imperative）**: 「どうやるか（How）」を逐次手順で記述。C, Java, Python等
- **宣言型（Declarative）**: 「何が欲しいか（What）」を記述し、手順は処理系に委ねる。SQL, HTML, CSS, Prolog等
- 宣言型言語は「プログラミング言語ではない」と言われがちだが、SQLやPrologは明確にプログラミング言語として扱われている
- **重要な示唆**: 命令型でなければプログラミング言語ではない、という主張は成り立たない

### 形式言語理論からの分類

- チョムスキー階層による言語の分類：
  - Type 0: 帰納的可算言語（チューリングマシン）
  - Type 1: 文脈依存言語
  - Type 2: 文脈自由言語（多くのプログラミング言語の構文）
  - Type 3: 正規言語（正規表現）
- CSSの構文はType 2（文脈自由文法）で記述可能
- ただし、構文の分類と「プログラミング言語であるか」は直接的に関係しない
  - 多くのプログラミング言語もType 2で構文定義されている

### 進化の方向性

CSSは一貫して「より表現力のある宣言的言語」へと進化している：
- **変数**: Custom Properties（`--var` / `var()`）
- **演算**: `calc()`, `min()`, `max()`, `clamp()`
- **条件分岐**: `@media`, `@supports`, `:has()`, Container Queries
- **型システム**: `@property` による型付きプロパティ
- **スコープ**: `@scope`, `@layer` によるカスケード制御
- **ネスト**: CSS Nesting

この進化は、CSSが「プログラミング的」な方向に進んでいることを示唆するとも言えるし、あくまで「宣言の表現力向上」であって本質は変わっていないとも解釈できる。

## 3. CSSはチューリング完全か？

### CSS + HTML（チェックボックスハック）でのRule 110

- **Rule 110** はスティーブン・ウルフラムが研究したセル・オートマトンの一種
- 2004年にMatthew CookがRule 110のチューリング完全性を証明
- 2011年にEli Fox-EpsteinがCSS + HTMLのみでRule 110を実装：
  - HTMLのチェックボックス（`<input type="checkbox">`）を状態記憶に使用
  - `:checked` 疑似クラスと隣接セレクタ（`~`, `+`）で状態遷移を記述
  - ユーザーがチェックボックスをクリックすることで「クロック」を供給

### 議論のポイント

| 主張 | 根拠 |
|------|------|
| **「チューリング完全である」派** | CSS + HTMLでRule 110を実装できた。CSSのセレクタとスタイル適用ルールで任意の計算を表現可能 |
| **「チューリング完全ではない」派** | ユーザー入力（クリック）が必要。CSS単体では状態遷移を自律的に進行できない。「外部クロック」に依存している |
| **中間的立場** | 「CSS + HTMLの組み合わせ」はチューリング完全に近い能力を持つが、厳密な意味でのチューリング完全性には議論の余地がある |

### 補足: CSS Animationsの自律的実行

- `@keyframes` とアニメーションの `animation-iteration-count: infinite` で自律的な状態変化は可能
- ただし、アニメーションは「事前に定義された状態の補間」であり、「任意の計算」ではない
- CSSのアニメーションだけでは分岐・ループ・メモリの要件を満たせない

### 結論的整理

> CSSが「チューリング完全か」という問いに対する回答は **「条件付きでYes」** が妥当。CSS + HTML（ユーザー入力込み）ではチューリング完全な計算を模倣できるが、CSS単体では不十分。ただし、これは「CSSがプログラミング言語かどうか」とは別の問いである（チューリング完全性はプログラミング言語の定義に必須ではない）。

## 4. CSSでできる「プログラミング的」なこと

### 4.1 演算: `calc()` と数学関数

```css
/* 四則演算 */
width: calc(100% - 2 * 20px);

/* 数学関数（CSS Values Level 4） */
font-size: clamp(1rem, 2.5vw, 2rem);
width: min(500px, 100%);
height: max(200px, 50vh);

/* 三角関数（CSS Values Level 4） */
transform: rotate(calc(sin(var(--angle)) * 1turn));

/* 指数・対数（提案中） */
/* pow(), sqrt(), log(), exp() 等 */
```

### 4.2 変数: Custom Properties

```css
:root {
  --primary-color: #3498db;
  --spacing-unit: 8px;
  --columns: 3;
}

.grid {
  grid-template-columns: repeat(var(--columns), 1fr);
  gap: calc(var(--spacing-unit) * 2);
}

/* スコープ付き変数（継承による） */
.dark-theme {
  --primary-color: #2980b9;
}
```

- Custom Propertiesは**カスケードと継承に従う**ため、CSSの文脈で自然に動作する
- JavaScriptから `setProperty()` で動的に変更可能
- **CSS単体での制限**: 変数の値を条件によって変えることが（`:has()` 登場以前は）困難だった

### 4.3 条件分岐

```css
/* メディアクエリ */
@media (max-width: 768px) {
  .sidebar { display: none; }
}

/* 機能クエリ */
@supports (display: grid) {
  .container { display: grid; }
}

/* :has() による条件分岐 */
.form:has(input:invalid) {
  border-color: red;
}

.card:has(img) {
  grid-template-rows: auto 1fr;
}

/* Container Queries */
@container sidebar (min-width: 300px) {
  .widget { flex-direction: row; }
}
```

### 4.4 ループ的な表現

```css
/* counter によるカウンティング */
ol {
  counter-reset: section;
}
li::before {
  counter-increment: section;
  content: counter(section) ". ";
}

/* nth-child による繰り返しパターン */
tr:nth-child(2n) { background: #f0f0f0; }
li:nth-child(3n+1) { color: red; }

/* @for 的なことはCSS単体では不可能 */
/* ただしSassではできる */
```

### 4.5 アニメーション・状態管理

```css
/* キーフレームアニメーション */
@keyframes pulse {
  0%, 100% { transform: scale(1); }
  50% { transform: scale(1.1); }
}

/* トランジション（状態間の補間） */
.button {
  transition: background-color 0.3s ease;
}
.button:hover {
  background-color: #2980b9;
}

/* チェックボックスハックによる状態管理 */
#toggle:checked ~ .menu {
  display: block;
}
```

### 4.6 「プログラミング的」の度合い

| 能力 | CSS単体 | CSS + HTML | Sass/LESS |
|------|---------|-----------|-----------|
| 変数 | ○（Custom Properties） | ○ | ○ |
| 演算 | ○（calc, min, max, clamp） | ○ | ○ |
| 条件分岐 | △（@media, @supports） | ○（:has, :checked） | ○（@if/@else） |
| ループ | △（counter, nth-child） | △ | ○（@for, @each, @while） |
| 関数定義 | ×（※ `@function` 提案あり） | × | ○（@function） |
| 再帰 | × | × | ○（制限あり） |
| I/O | × | × | ×（ビルド時のみ） |
| 状態管理 | △（:hover等） | ○（:checked等） | × |

## 5. 「CSSはプログラミング言語ではない」側の論拠

### 5.1 汎用的なロジック記述ができない

- CSSは「スタイリング」という特定の目的に特化しており、汎用的なアルゴリズムを記述する手段がない
- 再帰、任意のデータ構造、関数定義・呼び出しといった基本的なプログラミング構成要素が欠けている
- フィボナッチ数列の計算、ソートアルゴリズム、データ処理パイプラインなどは記述不可能

### 5.2 副作用がない

- CSSは純粋に「宣言」であり、外部状態を変更する能力がない
- ファイル操作、ネットワーク通信、データベースアクセスなどの副作用を起こせない
- これは純粋関数型言語の特性に似ているが、純粋関数型言語はモナド等で副作用を制御するのに対し、CSSにはそのような仕組みがない

### 5.3 I/O制御ができない

- ユーザー入力の読み取りは限定的（:hover, :focus, :checked等の擬似クラスのみ）
- 出力もビジュアルレンダリングに限定される
- 標準入出力、ファイルI/O、ネットワークI/Oの概念がない

### 5.4 制御フローがない

- 命令の逐次実行、条件分岐（if/else）、ループ（for/while）といった従来の制御フローが存在しない
- CSSの「ルール」は順序に関係なく宣言的に適用される（カスケードの順序はあるが、「実行順序」ではない）

### 5.5 チューリング完全ではない（CSS単体）

- 前述の通り、CSS単体ではチューリング完全性を示すことができていない
- これを「プログラミング言語ではない」根拠とする立場がある

## 6. 「CSSはプログラミング言語である」側の論拠

### 6.1 宣言的プログラミングのパラダイム

- 宣言型プログラミングは「プログラミング」の一形態として確立されている
- SQLは宣言型だがプログラミング言語として広く認知されている
- Haskellは純粋関数型（宣言的な側面が強い）だが間違いなくプログラミング言語
- **「命令型でなければプログラミングではない」は定義として狭すぎる**

### 6.2 DSL（ドメイン固有言語）としての見方

- CSSは「ビジュアルプレゼンテーション」というドメインに特化したDSL
- 正規表現、SQL、Make、AWK等もDSLであり、これらを「プログラミング言語ではない」とは通常言わない
- DSLは汎用プログラミング言語（GPL）より表現力が限定的であることが多いが、それは「プログラミング言語ではない」ことを意味しない

### 6.3 「プログラミング」の本質は何か

- プログラミングとは「コンピュータに対する指示を記述すること」
- CSSは明らかにコンピュータ（ブラウザのレンダリングエンジン）に対する指示を記述している
- ルールの競合解決（カスケード、詳細度計算）は複雑なアルゴリズムであり、その記述（=CSS）はプログラミングと見なせる

### 6.4 進化するCSSの表現力

- Custom Properties、calc()、:has()、Container Queries等の追加で、CSSの「プログラミング的」能力は年々向上
- CSS WGでは `if()` 関数や `@function` の提案も議論されている
- 「プログラミング言語ではない」と断言するための境界線は、CSSの進化によって年々曖昧になっている

### 6.5 HTMl/CSSとの比較

- HTMLも「プログラミング言語ではない」とよく言われるが、HTMLはマークアップ言語（構造の記述）
- CSSは「ルールに基づいてビジュアル出力を制御する」言語であり、HTMLよりも「プログラミング的」
- CSSのセレクタはパターンマッチング、カスケードは優先順位の解決アルゴリズムであり、計算的な側面がある

## 7. コミュニティの議論の歴史

### 7.1 「CSSは簡単」という偏見

- 2010年代前半まで、フロントエンド開発（特にCSS）は「本物のプログラミングではない」と見なされがちだった
- 背景：バックエンド・システムプログラミングを「高度」、フロントエンドを「デザイナーの仕事」と見る二分法
- この偏見はジェンダー問題とも関連：フロントエンド開発者に女性が多い傾向があり、軽視の背景に性差別的な要素があるという指摘もある

### 7.2 「Real Programmer」文化

- 「Real Programmers don't use Pascal」（1983年のエッセイ）に代表される、特定の言語やツールを使うことで「本物のプログラマー」かどうかを判定しようとする文化
- CSSやHTMLを「プログラミング言語ではない」と主張することが、ゲートキーピング（門番行為）に利用されるケースがある
- 「CSSができる」と言って面接で落とされた、という逸話はSNSで繰り返し共有されている

### 7.3 議論の変遷

| 時期 | 主な論調 |
|------|---------|
| 2000年代 | CSSは「デザインツール」。プログラミング言語かどうかの議論自体が少ない |
| 2010年代前半 | Sass/LESSの普及で「CSSに足りないもの」が可視化。CSSはプログラミング言語ではないが、Sassはプログラミング的 |
| 2010年代後半 | CSS Custom Properties登場。「CSSも変数が使えるようになった」。議論が活発化 |
| 2020年代 | :has(), Container Queries, Nesting等で表現力が急激に向上。「CSSはもはやプログラミング言語と言えるのでは」という声が増加 |

### 7.4 有名な発言・記事

- **Chris Coyier（CSS-Tricks創設者）**: 「CSSは確かにプログラミング言語だ」と主張。CSSの複雑さと深さを繰り返し発信
- **Lara Schenck**: 「CSS Is, In Fact, a Programming Language」（2019年）。CSSが宣言的プログラミング言語であることを体系的に論じた記事
- **Eric Meyer**（CSS権威）: CSSの表現力と複雑さを長年にわたり発信。「CSSを軽視するな」というメッセージ
- **多くのTwitter/X議論**: 「Is CSS a programming language?」は定期的にバイラルになるトピック。毎年数回、大規模な議論が発生

## 8. CSS拡張言語（プリプロセッサ）

### Sass（2006年〜）

- Hampton Catlinが開発、Natalie Weizenbaum（nex3）がメイン開発者に
- **SCSS構文**（CSS互換）と**Sass構文**（インデントベース）の2種類
- 提供する「プログラミング的」機能：
  - 変数（`$variable`）
  - ネスト
  - ミックスイン（`@mixin` / `@include`）
  - 関数定義（`@function` / `@return`）
  - 条件分岐（`@if` / `@else if` / `@else`）
  - ループ（`@for`, `@each`, `@while`）
  - リスト・マップ操作
- Sassの存在は「CSSに何が足りないか」を明確にした
- 逆に言えば、Sassが必要だった事実は「CSSだけではプログラミング的な記述が不十分」という証拠ともなる

### LESS（2009年〜）

- Alexis Sellierが開発
- JavaScriptベースで動作（ブラウザでもNode.jsでも）
- Sassに近い機能を提供するが、やや簡素
- Bootstrap 3がLESSを採用したことで普及（Bootstrap 4でSassに移行）

### PostCSS（2013年〜）

- プリプロセッサではなく「CSSトランスフォーマー」
- プラグインアーキテクチャで拡張可能
- 代表的プラグイン：
  - **Autoprefixer**: ベンダープレフィックス自動付与
  - **cssnano**: CSS圧縮
  - **postcss-preset-env**: 将来のCSS構文を現在のブラウザで使用可能に
- PostCSSの存在は「CSSのエコシステムにはビルドツールが必要」という認識を広めた

### ネイティブCSSへの還元

- Sass/LESSの人気機能は徐々にネイティブCSSに取り込まれている：
  - 変数 → CSS Custom Properties（2017年）
  - ネスト → CSS Nesting（2023年）
  - 数学関数 → calc(), min(), max(), clamp()
  - 色関数 → color-mix(), relative color syntax
- この傾向は「CSSがプログラミング言語に近づいている」とも解釈できる

## 10. 結論的な視点：グラデーションとしての「プログラミング言語」

### 二項対立ではなくスペクトラム

「プログラミング言語であるか否か」は二者択一の問いではなく、**スペクトラム（グラデーション）** として捉えるのが適切：

```
HTMLのみ ← CSSのみ ← CSS + HTML ← Sass/LESS ← SQL ← Haskell ← Python ← C
（マークアップ）  （宣言的スタイリング）  （宣言的+状態管理）  （宣言型DSL）  （汎用命令型）

         宣言的 ──────────────────────────────────────────── 命令的
         限定的 ──────────────────────────────────────────── 汎用的
```

### 定義次第で答えが変わる

| 定義 | CSSは？ |
|------|--------|
| チューリング完全な言語 | ×（CSS単体）/ △（CSS + HTML） |
| コンピュータへの指示を記述する言語 | ○ |
| ドメイン固有言語（DSL） | ○ |
| 宣言的プログラミング言語 | ○（広義には） |
| 汎用プログラミング言語 | × |

### 台本で使える核心メッセージ

1. **「プログラミング言語」に唯一の定義はない** ― 何を基準にするかで答えが変わる
2. **CSSは進化し続けている** ― 2024年のCSSは1996年のCSSとは全く別物。変数、演算、条件分岐の要素を持つ
3. **この議論の本質は技術的というより文化的** ― 「CSSはプログラミング言語ではない」という主張がゲートキーピングに使われてきた歴史がある
4. **重要なのは「何と呼ぶか」ではなく「何ができるか」** ― CSSを書く人がプログラマーかどうかはラベルの問題であって、能力の問題ではない
5. **グラデーション的理解が健全** ― 白か黒かではなく、プログラミング的な性質を持つ宣言的言語として理解するのが最も正確

### 注意点

- **煽り・攻撃的な切り口NG**。「CSSもわからないのにプログラマーを名乗るな」的な論調は避ける
